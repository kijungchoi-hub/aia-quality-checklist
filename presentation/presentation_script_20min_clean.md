# AI 품질 체크리스트 20분 발표 대본

## 발표 제목

프로젝트 전체 관점의 AI 품질 체크리스트 체계 소개

## 발표 도입

안녕하세요.  
오늘은 프로젝트 전체 관점에서 AI 품질 체크리스트를 어떻게 보고, 왜 이 문서 체계가 필요한지 공유드리겠습니다.

오늘 발표는 세 가지를 중심으로 말씀드리겠습니다.

1. 왜 AI 품질을 개별 기능이 아니라 체크리스트 체계로 관리해야 하는지
2. 현재 정리된 문서 구조와 체크리스트가 실제로 무엇을 다루는지
3. 이 체크리스트를 프로젝트 실행과 개선활동에 어떻게 연결할지

## 1. 왜 체크리스트 체계가 필요한가

먼저 배경부터 말씀드리겠습니다.

AI 프로젝트는 일반 소프트웨어와 다르게, 기능이 동작한다고 해서 품질이 확보됐다고 보기 어렵습니다.  
같은 질문에도 답변 품질이 흔들릴 수 있고, 데이터가 바뀌거나 프롬프트가 바뀌거나 검색 구성이 바뀌면 이전에 잘 되던 결과가 다시 나빠질 수 있습니다.

특히 생성형 AI나 RAG 기반 서비스는 문제 원인이 한 군데에만 있지 않습니다.  
데이터 문제일 수도 있고, 검색 문제일 수도 있고, 프롬프트 문제일 수도 있고, 모델 문제일 수도 있습니다.

그래서 결과만 보고 "AI가 별로다", "모델을 바꿔야 한다"라고 접근하면 개선활동이 반복되기 어렵습니다.

이 프로젝트에서 정리한 핵심은 바로 그 지점입니다.  
AI 품질은 단일 모델의 성능 문제가 아니라, 데이터, 검색, 컨텍스트, 프롬프트, 평가, 회귀 점검, 보안과 정책, 그리고 개선활동 기록까지 포함하는 체계로 관리해야 한다는 점입니다.

그래서 이번 문서 정리의 목적은 체크리스트를 예쁘게 정리하는 데 있지 않습니다.  
실제로 프로젝트가 바뀌어도 재사용할 수 있는 품질 기준을 만들고, 개선활동이 누적될 수 있는 구조를 만드는 데 있습니다.

## 2. 문서 구조 설명

이제 문서 구조를 기준으로 전체를 설명드리겠습니다.

현재 문서는 크게 네 묶음으로 구성돼 있습니다.

- [`checklists`](../README.md)
- [`projects`](ai_rag_service_project_tracker.md)
- [`governance`](../governance/README.md)
- [`records`](../records/quality_checklist_overview.md)

`checklists`는 기준 문서입니다.  
무엇을 점검해야 하는지 정의합니다.

`projects`는 실행 관리 문서입니다.  
체크리스트 항목을 실제 프로젝트 단위의 일정, 우선순위, 담당, 진행률로 연결합니다.

`governance`는 기준서입니다.  
실패 분류 기준, 배포 게이트, 평가 기록, 확장 검증처럼 일관된 판단 기준을 담습니다.

`records`는 개선활동 기록 영역입니다.  
주간 점검 기록, 실험 설계, 회귀 테스트, 실패 사례 정리처럼 실제 개선활동의 흔적을 남기는 곳입니다.

이 구조를 한 문장으로 정리하면 이렇습니다.  
체크리스트로 기준을 잡고, 프로젝트 문서로 실행하고, 기준서로 판단을 맞추고, 기록 문서로 개선활동을 축적한다.

## 3. 체크리스트 세 종류

다음으로 체크리스트 자체를 설명드리겠습니다.

현재 체크리스트는 세 종류로 나뉘어 있습니다.

- [`answer_checklist.md`](../checklists/answer_checklist.md)
- [`summary_checklist.md`](../checklists/summary_checklist.md)
- [`ai_rag_service_checklist.md`](../checklists/ai_rag_service_checklist.md)

답변 품질 체크리스트는 가장 직접적인 응답 품질을 다룹니다.  
평가 기준 정합성, 검색 품질, 프롬프트와 생성 품질, 컨텍스트 전략, 모델 튜닝, 오류 분석과 개선관리까지 다룹니다.

요약 품질 체크리스트는 요약 서비스에 맞는 항목을 다룹니다.  
과답, 누락, 왜곡 같은 요약 특유의 문제를 관리하고, 평가 자동화와 요약 구조 개선까지 포함합니다.

AI RAG 서비스 체크리스트는 조금 더 상위 개념입니다.  
특정 프로젝트가 아니라, 일반적인 AI + RAG 서비스가 공통으로 점검해야 할 내용을 정리합니다.  
데이터 준비, 검색 전략, 컨텍스트 구성, 생성 통제, 모델 선택, 평가 체계, 실패 분류, 보안, 관측성, 최종 점검까지 포괄합니다.

중요한 점은 이 세 문서가 서로 따로 노는 문서가 아니라는 것입니다.  
답변 품질 체크리스트와 요약 품질 체크리스트는 업무 유형에 맞춘 상세 기준이고, AI RAG 서비스 체크리스트는 그 위에 놓이는 공통 프레임입니다.

즉, 프로젝트를 시작할 때는 이 셋 중 하나만 고르는 것이 아니라, 공통 기준은 RAG 체크리스트에서 보고, 업무 유형별 상세 기준은 답변 또는 요약 체크리스트에서 보는 방식으로 쓰는 것이 맞습니다.

## 4. 프로젝트 관리표의 역할

그다음 중요한 것은, 체크리스트를 그냥 읽고 끝내지 않는 것입니다.

체크리스트는 질문 목록일 뿐입니다.  
실제 품질 개선은 무엇이 미착수인지, 무엇이 우선순위가 높은지, 어떤 항목을 이번 사이클에서 다룰지가 정리돼야 시작됩니다.

그래서 [`ai_rag_service_project_tracker.md`](ai_rag_service_project_tracker.md) 같은 프로젝트 관리표가 필요합니다.

프로젝트 관리표에서는 각 항목을 우선순위, 상태, 담당자, 진행률, 메모 기준으로 관리합니다.  
이 문서의 의미는 단순한 진행표가 아닙니다. 체크리스트를 실행 가능한 일로 바꾸는 연결 지점입니다.

예를 들어 검색 실패 원인을 분류한다는 체크리스트 항목이 있으면, 프로젝트 관리표에서는 이 항목을 누가 맡는지, 언제까지 할지, 어떤 산출물이 남아야 하는지 적습니다.

회귀 테스트 기준을 마련한다는 항목이 있으면, 이것도 마찬가지로 실제 샘플 구축, 결과 기록, 배포 게이트 연결까지 실행 단위로 분해됩니다.

이렇게 되면 체크리스트는 선언이 아니라 실행 계획으로 바뀝니다.

## 5. Governance 문서의 역할

다음으로 governance 문서의 역할을 말씀드리겠습니다.

AI 프로젝트에서는 판단 기준이 흔들리면 팀마다 다른 말을 하게 됩니다.  
누군가는 검색이 문제라고 하고, 누군가는 모델이 문제라고 하고, 누군가는 평가셋이 잘못됐다고 말합니다.

이 자체는 자연스럽지만, 같은 기준으로 분류하고 기록하지 않으면 개선활동이 누적되지 않습니다.

그래서 governance 문서가 필요합니다.

여기에는 대표적으로 다음 문서들이 있습니다.

- [`quality_operating_framework.md`](../governance/quality_operating_framework.md)
- [`search_failure_codebook.md`](../governance/search_failure_codebook.md)
- [`generation_failure_casebook.md`](../governance/generation_failure_casebook.md)
- [`manual_auto_review_log.md`](../governance/manual_auto_review_log.md)
- [`release_gate_checklist.md`](../governance/release_gate_checklist.md)

검색 실패 코드북은 검색 실패를 어떤 유형으로 나눌지 정의합니다.  
색인 문제인지, 메타데이터 문제인지, 질의 해석 문제인지, 랭킹 문제인지 구분합니다.

생성 실패 케이스북은 검색은 성공했는데 답변이 틀리는 경우를 별도로 다룹니다.  
조건 누락인지, 근거 왜곡인지, 환각인지 분리해서 봅니다.

수동 및 자동 평가 기록은 자동 점수와 사람 판단을 한 자리에서 보게 해 줍니다.  
자동 점수만 믿지 않고, 그렇다고 수동 검수만으로 끌고 가지도 않도록 균형을 잡는 문서입니다.

배포 게이트 체크리스트는 마지막 안전장치입니다.  
이 개선활동을 실제 반영해도 되는지, 회귀는 없는지, 주요 리스크는 무엇인지 판단하는 기준입니다.

이 문서들이 있어야 무엇이 실패했고, 왜 실패했고, 어떤 기준으로 보류하거나 반영했는지가 남게 됩니다.

## 6. Records 문서의 역할

다음은 records, 즉 개선활동 기록 영역입니다.

여기가 실무에서 가장 중요합니다.  
체크리스트와 기준서가 있어도, 실험과 검증 기록이 쌓이지 않으면 다음 사람이 이어받을 수 없습니다.

이 영역에는 다음과 같은 문서가 있습니다.

- [`quality_checklist_overview.md`](../records/quality_checklist_overview.md)
- [`experiment_design.md`](../records/templates/operations/experiment_design.md)
- [`weekly_quality_log.md`](../records/templates/logs/weekly_quality_log.md)
- [`regression_test_tracker.md`](../records/templates/validation/regression_test_tracker.md)

실험 설계서는 무엇을 바꾸는지보다 왜 바꾸는지를 남기는 문서입니다.  
가설, 변경 전 상태, 변경 후 기대 결과, 리스크, 검증 기준을 정리합니다.

주간 품질 점검 기록은 단순 주간 보고서가 아닙니다.  
이번 주에 어떤 개선활동을 했고, 무엇이 좋아졌고, 무엇이 리스크인지, 다음 주에 무엇을 할지를 같은 형식으로 남깁니다.

회귀 테스트 관리표는 잘 되던 것이 나빠지지 않았는지 확인하는 문서입니다.  
AI는 개선보다 회귀 관리가 더 어려운 경우가 많기 때문에 이 문서가 중요합니다.

이 문서 구조가 갖는 가장 큰 장점은 사람이 바뀌어도 프로젝트 맥락이 남는다는 점입니다.

## 7. 실제 적용 흐름

이제 실제 적용 흐름을 말씀드리겠습니다.

1. 프로젝트가 시작되면 먼저 데이터 원본을 확인합니다.
2. 답변형인지, 요약형인지, 공통 RAG형인지 판단합니다.
3. 해당 체크리스트를 선택합니다.
4. 프로젝트 관리표에 이번 사이클에서 다룰 항목을 옮기고 우선순위를 정합니다.
5. governance 문서로 판단 기준을 맞춥니다.
6. records 영역에서 실험 설계, 결과 기록, 회귀 확인, 주간 점검 기록을 남깁니다.
7. 마지막으로 배포 또는 개선 반영 전에 게이트 체크리스트로 최종 판단합니다.

이렇게 보면 전체 흐름은 단순합니다.  
기준 선택, 실행 계획, 판단 기준 정렬, 개선활동 기록, 최종 점검입니다.

## 발표 마무리

이제 마무리하겠습니다.

이번 AI 품질 체크리스트 체계의 핵심은 문서를 많이 만드는 것이 아닙니다.  
품질 문제를 같은 언어로 보고, 같은 기준으로 분류하고, 같은 형식으로 개선활동을 누적하는 것입니다.

정리하면 세 가지입니다.

1. AI 품질은 모델 하나가 아니라 데이터, 검색, 생성, 평가, 회귀, 정책까지 함께 봐야 합니다.
2. 체크리스트는 기준 문서이고, 프로젝트 관리표와 기준서, 기록 문서가 붙어야 실제 개선활동이 됩니다.
3. 이 체계가 있어야 프로젝트 단위의 시행착오가 조직의 공통 자산으로 바뀝니다.

오늘 공유드린 문서는 단일 프로젝트 문서가 아니라, 앞으로 다른 AI 과제에도 재사용할 수 있는 품질 기준과 개선활동 구조라고 이해해 주시면 됩니다.

이상으로 발표를 마치겠습니다.  
질문 받겠습니다.
