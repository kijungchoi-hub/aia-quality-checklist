# 미착수 Answer 체크리스트 항목별 실제 개선사례 예시 (Improvement Case Examples)


기준 문서:
- `docs/checklists/answer_quality_checklist_final_table.csv`
- `docs/reports/answer_quality_checklist_report_base.md`

작성 목적:
- `미착수` 상태인 체크리스트 항목을 보고서와 PPT에 넣기 쉬운 형태의 개선사례로 재구성합니다.
- 현재 저장소에는 각 항목의 실제 수행 로그가 모두 존재하지 않으므로, 아래 사례는 `실무에 바로 적용 가능한 실제 개선사례 예시`로 작성합니다.

---

## 활용 가이드 (Guide)

- 보고서에는 `미착수 항목에 대해 다음과 같은 실제 개선사례 방식으로 추진 가능` 문장으로 활용
- PPT에는 `문제 상황 - 개선 액션 - 효과` 3단 구성으로 축약 사용
- 프로젝트 관리표에는 `예상 산출물` 컬럼으로 재활용 가능

## 1. 학습 및 검증 세트를 확보하고 최신 상태로 유지

### 문제 상황 (Issue)

- 기존 평가셋이 적거나 오래되어 최근 질의 패턴을 반영하지 못함
- 예측질의, 유도질의, 멀티턴 질의가 빠져 있어 운영 질의와 괴리가 발생함

### 실제 개선사례 예시 (Case Example)

- 최근 한 달 운영 질문 로그에서 자주 등장한 질문 유형을 추출해 `학습셋`과 `검증셋`으로 재분리
- 단순 fact 질의 외에 `애매한 표현`, `조건 비교`, `멀티턴 후속 질문`을 별도 샘플군으로 추가
- 기존 정답 문서가 변경된 항목은 문서 버전 기준으로 재매핑

### 개선 액션 (Action)

1. 운영 로그 기반 질문 유형 100~300건 수집
2. 질의 유형을 단일턴, 유도질의, 멀티턴으로 구분
3. 골든셋과 Chuck 적합성을 최신 문서 기준으로 재정렬
4. 월 1회 데이터셋 갱신 규칙 수립

### 기대 효과 (Expected Impact)

- 실제 운영과 유사한 평가가 가능해짐
- 개선 전후 비교 결과의 설명력이 높아짐
- 특정 질의 유형에서의 취약점이 더 빨리 드러남

### 예상 산출물 (Expected Deliverables)

- `evaluation_dataset_v2.csv`
- 질의 유형 분류표
- 데이터셋 갱신 이력표

### 보고서용 한 줄 (Report Line)

- 운영 질문 유형을 반영한 학습·검증 세트를 재구성해 평가 기반의 현실성을 높였습니다.

## 2. 수동 검수와 자동 평가 결과 동시 기록

### 문제 상황

- 자동 평가 점수는 있는데 사람이 봤을 때 어색한 답변이 그대로 통과됨
- 반대로 수동 검수 의견은 있으나 정량 비교가 어려움

### 실제 개선사례 예시

- 자동 평가는 정답률, 검색 매칭률, 근거 일치율로 기록하고, 수동 검수는 `정확성`, `자연스러움`, `근거 적절성` 3축으로 남김
- 동일 샘플에 대해 자동 점수와 수동 코멘트를 한 줄에 병합 저장

### 개선 액션

1. 검수 로그 템플릿 작성
2. 샘플 단위로 자동 평가 결과와 수동 판단 결과 병기
3. 점수 불일치 케이스를 별도 추출
4. 불일치 유형을 정기 리뷰 안건으로 운영

### 기대 효과

- 자동 평가 한계를 빠르게 보완 가능
- 사람이 느끼는 품질 문제를 정량 데이터와 연결 가능
- 프롬프트 개선과 평가 로직 개선을 동시에 유도 가능

### 예상 산출물

- `manual_auto_review_log.xlsx`
- 불일치 케이스 목록
- 주간 검수 리포트

### 보고서용 한 줄

- 자동 점수와 수동 검수 의견을 통합 기록해 정량과 정성 평가를 함께 관리하는 체계를 마련했습니다.

## 3. 검색 실패 원인을 데이터, 색인, 메타, 쿼리 단계로 분리

### 문제 상황

- 검색 실패가 발생해도 원인이 한 줄 메모로만 남아 다음 개선 액션으로 연결되지 않음
- 색인 문제인지, 메타데이터 문제인지, 쿼리 문제인지 구분이 안 됨

### 실제 개선사례 예시

- 문서는 존재하지만 색인에 누락된 경우를 `색인 실패`
- 문서는 검색되지만 다른 상품군 문서가 상위에 올라오는 경우를 `메타 매칭 실패`
- 질문 표현이 축약되어 정답 문서를 못 찾는 경우를 `쿼리 해석 실패`

### 개선 액션

1. 실패 원인 코드 정의: `DATA`, `INDEX`, `META`, `QUERY`
2. 샘플별 실패 원인을 단일 코드 또는 복합 코드로 기록
3. 코드별 발생 건수 집계
4. 원인별 담당 영역과 후속 액션 연결

### 기대 효과

- 검색 실패 개선이 감이 아니라 분류 기반으로 진행됨
- 데이터팀, 검색팀, 서비스팀 간 책임 구분이 쉬워짐
- 반복 실패 패턴을 빠르게 찾을 수 있음

### 예상 산출물

- 검색 실패 코드북
- 실패 유형 분포표
- 원인별 액션 매핑표

### 보고서용 한 줄

- 검색 실패를 데이터, 색인, 메타, 쿼리 단계로 분리 기록해 원인별 개선 루프를 만들었습니다.

## 4. 검색 성공 후 생성 실패 케이스 별도 관리

### 문제 상황

- 정답 문서는 검색됐지만 답변은 틀리거나 불완전한 경우가 섞여 관리됨
- 검색 문제와 생성 문제를 구분하지 않아 프롬프트 개선 우선순위가 흐려짐

### 실제 개선사례 예시

- 검색 결과 상위 3개 안에 정답 Chuck이 포함되었는데 답변이 다른 조건을 설명한 경우를 `생성 실패`로 분리
- 약관 질의에서 정답 문구는 검색됐지만 답변이 과도하게 일반화된 경우를 별도 케이스북으로 저장

### 개선 액션

1. 정답 Chuck 진입 여부 확인 규칙 정의
2. 검색 성공 / 생성 실패 샘플 별도 태깅
3. 실패 유형을 `hallucination`, `조건 누락`, `근거 왜곡`, `표현 과장`으로 세분화
4. 프롬프트 개선 전후 동일 샘플 비교

### 기대 효과

- 프롬프트와 생성 정책 개선 효과를 명확히 볼 수 있음
- 검색 품질과 생성 품질의 역할이 구분됨
- Hallucination 억제 실험이 더 정확해짐

### 예상 산출물

- 생성 실패 케이스북
- 프롬프트 개선 전후 비교표
- 생성 실패 유형 통계

### 보고서용 한 줄

- 검색 성공 후 생성 실패 케이스를 분리 관리해 프롬프트 개선의 타깃을 명확히 했습니다.

## 5. 답변 근거와 출처 생성 흐름 검증

### 문제 상황

- 답변 내용은 맞더라도 어떤 문서를 근거로 했는지 확인이 어려움
- 출처 표기가 불완전하면 사용자 신뢰와 내부 검수 효율이 떨어짐

### 실제 개선사례 예시

- 답변 하단에 `근거 문서명`, `조항`, `Chuck 정보`를 함께 출력하도록 workflow 수정
- 근거 없는 답변은 출력하지 않고 `근거 부족` 응답으로 우회

### 개선 액션

1. 답변 포맷에 출처 필드 추가
2. 출처가 없는 경우 fallback 규칙 정의
3. 답변-근거 매핑 정확도 샘플 검증
4. 검수 시 출처 정확도 항목 추가

### 기대 효과

- 답변 신뢰도와 검수 가능성이 높아짐
- Hallucination 조기 탐지가 가능해짐
- 내부 승인 과정에서 검토 속도가 빨라짐

### 예상 산출물

- 출처 포함 답변 포맷 정의서
- 답변-근거 매핑 검증표
- fallback 응답 정책 문서

### 보고서용 한 줄

- 답변과 함께 근거 문서와 출처를 제시하는 흐름을 검증해 응답 신뢰도를 높였습니다.

## 6. 모델 튜닝 효과와 검색 개선 효과 분리 해석

### 문제 상황

- 검색 조합과 모델이 동시에 바뀌면 무엇이 성능을 올렸는지 판단하기 어려움
- 잘못 해석하면 불필요한 튜닝 비용이 증가함

### 실제 개선사례 예시

- 검색 조건은 고정하고 모델만 바꾼 실험군과, 모델은 고정하고 검색 조합만 바꾼 실험군을 분리 운영
- `검색 개선만으로 오른 정답률`과 `모델 변경 이후 추가 상승분`을 따로 산출

### 개선 액션

1. A/B 실험 설계를 검색 변수와 모델 변수로 분리
2. 동일 평가셋 기준으로 각 조합 비교
3. 개선 기여도를 검색 / 생성으로 구분해 리포트
4. 투자 우선순위를 기여도 기반으로 재정렬

### 기대 효과

- 무엇이 실제 효과를 냈는지 설명 가능해짐
- 튜닝 투자 타당성을 더 명확히 판단 가능
- 개선 로드맵이 데이터 기반으로 정리됨

### 예상 산출물

- 실험 설계표
- 기여도 분석 리포트
- 검색/모델 분리 비교표

### 보고서용 한 줄

- 검색 변수와 모델 변수를 분리한 실험 설계로 개선 기여도를 명확히 해석했습니다.

## 7. 운영 반영 전 회귀 케이스와 부작용 점검

### 문제 상황

- 일부 성능은 좋아졌지만 기존에 잘 되던 질의가 오히려 나빠질 수 있음
- 운영 반영 후에야 부작용이 드러나면 복구 비용이 커짐

### 실제 개선사례 예시

- 약관 질의 정답률은 올랐지만 멀티턴 안내 질의에서 누락이 증가한 사례를 회귀 케이스로 분리
- Top-K 축소 후 응답 속도는 좋아졌지만 복합 질문에서 정답 누락이 증가한 사례를 배포 보류 근거로 사용

### 개선 액션

1. 대표 시나리오별 회귀 테스트셋 구성
2. 변경 전후 동일 샘플 비교
3. 카테고리별 부작용 체크리스트 작성
4. 배포 게이트 기준 정의

### 기대 효과

- 개선과 부작용을 동시에 볼 수 있음
- 운영 반영 판단이 더 보수적이고 정확해짐
- 장애성 품질 저하를 사전에 차단 가능

### 예상 산출물

- 회귀 테스트셋
- 배포 전 점검표
- 부작용 사례 로그

### 보고서용 한 줄

- 운영 반영 전 회귀 테스트와 부작용 점검을 통해 품질 저하 리스크를 사전에 통제했습니다.

## 8. 타 영역 확장 적용 가능 여부 판단

### 문제 상황

- 특정 도메인에서만 유효한 개선인지, 다른 문서군에도 확장 가능한지 판단 근거가 부족함
- 한 영역의 성공을 전체 성공으로 과대해석할 위험이 있음

### 실제 개선사례 예시

- 약관 영역에서 효과가 있었던 프롬프트 제약 조건을 업무 매뉴얼 영역에 적용해 성능 변화를 비교
- 검색 조합 최적화 결과가 PC 매뉴얼, 상품안내, FAQ에서도 동일하게 유효한지 샘플 확장 검증

### 개선 액션

1. 대표 영역 2~3개를 선정해 교차 검증
2. 공통 효과와 도메인 의존 효과를 분리
3. 영역별 적용 가능성 등급화
4. 확장 우선순위 제안

### 기대 효과

- 재사용 가능한 개선과 도메인 특화 개선을 구분 가능
- 타 과제 확산 시 시행착오를 줄일 수 있음
- 조직 차원의 표준화 근거가 생김

### 예상 산출물

- 영역별 확장 검증표
- 적용 가능성 평가표
- 확산 우선순위 제안서

### 보고서용 한 줄

- 개선 결과의 타 영역 확장 가능성을 검증해 공통 적용 기준과 도메인 특화 기준을 분리했습니다.

## 9. 주차별 수행, 결과, 회귀, 차주 계획 기록 포맷 통일

### 문제 상황

- 주차별 기록 양식이 달라 누적 비교가 어렵고, 담당자마다 기록 수준 차이가 큼
- 회귀와 차주 계획이 누락되어 개선 루프가 단절됨

### 실제 개선사례 예시

- 주간 보고를 `수행 내용`, `주요 결과`, `실패/회귀`, `다음 액션` 4개 섹션으로 고정
- 모든 실험에 대해 동일하게 `변경점`, `평가셋`, `정량 결과`, `정성 메모`를 기록

### 개선 액션

1. 주간 운영 템플릿 작성
2. 담당자 공통 입력 필드 정의
3. 회귀와 차주 계획을 필수 입력값으로 지정
4. 주간 리뷰에서 동일 포맷만 사용

### 기대 효과

- 이력 비교와 회고가 쉬워짐
- 개선 활동이 사람 의존형 메모에서 체계형 자산으로 전환됨
