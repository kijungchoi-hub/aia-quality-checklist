## 1. 보고 개요 (Overview)

본 보고서는 `answer` 품질 개선 활동을 주요 학습 사항, 실행 과제, 공유 및 확산 관점에서 재구성한 문서입니다.
본 문서의 목적은 개별 관리 항목을 단순 나열하는 데 있지 않습니다.
데이터, 검색, 생성, 운영이 상호 연결된 구조 안에서 어떤 목표 상태를 만들어야 하는지와 이를 위해 무엇을 우선 실행해야 하는지를 체계적으로 설명하는 데 있습니다.

이번 활동을 통해 확인한 핵심 사항은 다음과 같습니다.
`LLM + RAG` 기반 생성형 AI 서비스에서 최종 답변 품질 개선 작업은 단일 기능 개선으로 해결되지 않습니다.
데이터 정합성, 검색 회수율, 컨텍스트 구성, 생성 안정성, 운영 기준은 서로 영향을 줍니다.
이 때문에 개별 개선 작업의 복합도가 높습니다.
특정 결과가 왜 개선되었는지 또는 왜 악화되었는지 원인을 분리해 해석하기도 어렵습니다.

따라서 답변 품질 개선의 본질은 단순한 모델 교체가 아니다.
신뢰 가능한 평가 기반 확보, 검색과 생성의 분리 개선, 그리고 이를 운영 가능한 관리 체계로 축적하는 데 있습니다.
더 나아가 본 문서는 개별 프로젝트 대응을 넘어, 표준화 관점에서 체계적인 품질관리를 위한 방법을 제시하는 기준 문서로 활용될 수 있습니다.

## 1.1 핵심 제안 (Core Proposal)

본 보고서의 핵심 제안은 다음과 같습니다.

1. 관리 기준을 `체크리스트 항목 수`에서 `목표 결과 확보 여부`로 전환합니다.
2. 품질 문제를 `검색 실패`와 `생성 실패`로 분리하여 원인 분석 정밀도를 높입니다.
3. 개선 활동을 `평가 기준`, `실험 기록`, `회귀 기준`, `배포 게이트`와 연결합니다.
4. 개선 경험을 공통 코드북과 운영 템플릿으로 표준화하여 조직 자산으로 축적합니다.

## 2. 추진 배경 (Background)

기존의 답변 품질 개선 활동은 데이터, 검색, 생성, 운영 영역에 분산되어 관리되어 왔습니다.
이로 인해 동일한 문제를 서로 다른 표현으로 반복 관리하거나, 개별 실험 결과를 운영 반영 판단과 직접 연결하기 어려운 한계가 존재하였습니다.
또한 회귀 추적 및 배포 기준이 충분히 정립되지 않아, 개선 활동이 실제 운영 품질 향상으로 이어지는 경로가 명확하지 않았습니다.

특히 품질 저하 현상은 단일 원인으로 설명되기 어렵습니다.
검색 회수율 저하, 근거 누락, 답변 왜곡, 운영 기준 미비는 각각 별개처럼 보이지만 실제 사용자 경험에서는 하나의 품질 문제로 나타난습니다.
따라서 기능 단위가 아니라 서비스 흐름 단위로 품질 문제를 해석하고 정리할 공통 프레임이 필요하였습니다.

또한 `LLM + RAG` 서비스에서는 검색 개선이 생성 품질에 영향을 주고, 프롬프트 변경이 근거 표현과 회귀 결과에 영향을 주며, 평가 기준 변경이 개선 효과 해석 자체를 바꿀 수 있습니다.
즉, 개선 작업 간 상호영향이 크기 때문에 개별 작업만 보고는 전체 품질 개선 효과를 설명하기 어렵습니다.

이에 따라 본 정리는 실행 가능한 공통 프레임을 수립하는 데 초점을 두었습니다.
기존 관리 항목을 목표 결과 중심으로 재구성하고, 주요 학습 사항, 실행 과제, 공유 및 확산 관점으로 구조화함으로써 개별 실험 중심의 접근을 운영 가능한 품질 관리 체계로 전환하고자 하였습니다.
동시에 이는 표준화 관점에서 조직이 공통으로 활용할 수 있는 품질관리 방법을 제시하기 위한 목적도 가집니다.

## 3. 목표 결과 중심 현황 (Outcome Status)

본 절에서 중요한 것은 관리 대상의 수량이 아니라, 최종적으로 어떤 품질 결과를 확보해야 하는가입니다.
따라서 현황은 진행상태 중심이 아니라 목표 상태와 개선점 중심으로 해석해야 합니다.

현재 확인된 핵심 리스크는 아래 목표 결과가 아직 충분히 확보되지 않았다는 점이며, 이는 곧 품질 개선의 우선 조치 영역을 의미합니다.

| 목표 결과 (Target Outcome) | 개선점 (Improvement Point) | 개선방향 (Improvement Direction) | 판단 기준 (Decision Criteria) |
|---|---|---|---|
| 신뢰 가능한 평가 기반 확보 | 평가셋 정합성과 기준 통일이 완전하지 않음 | 검증 세트 최신화, 정답지/Chuck 적합성 보강 | 동일 샘플 전후 비교 가능 |
| 검색 품질의 원인 분해 가능 | 검색 실패 유형이 체계적으로 누적되지 않음 | 실패 코드북 정비, 검색 조합 비교 실험 | 실패 원인 분류 기준 정착 |
| 생성 품질의 안정적 통제 | 검색 성공 후 생성 실패가 별도 관리되지 않음 | 생성 실패 케이스북 운영, 출처 흐름 검증 | 생성 실패 유형 별도 집계 가능 |
| 운영 반영 판단의 일관성 확보 | 회귀 기준과 배포 게이트가 충분히 정착되지 않음 | 회귀 테스트셋, 배포 게이트 기준 운영 | 배포 승인/보류 사유 설명 가능 |

따라서 향후 관리 방향은 "몇 개 항목을 완료했는가"보다 "어떤 목표 상태가 확보되었는가"를 기준으로 전환할 필요가 있습니다.
실무적으로는 각 결과 상태별 개선점이 얼마나 해소되었는지와, 개선방향이 실제 운영 체계로 연결되고 있는지를 중심으로 추적하는 것이 적절합니다.

## 4. 주요 학습 사항 (Key Learnings)

### 4.1 데이터 및 평가 기준 (Data & Evaluation)

첫 번째 학습 사항은 데이터와 평가 기준의 정합성이 모든 품질 개선 활동의 전제 조건이라는 점입니다.
골든셋, 정답지, Chuck 적합성이 확보되지 않으면 Baseline 정확도와 실패 사례를 신뢰성 있게 해석할 수 없습니다.
검색 개선 효과와 모델 개선 효과를 구분하는 것 또한 어렵습니다.

또한 정답률, 오답률, 소답률, 과답률, 검색 매칭률 등 주요 지표의 정의를 통일해야 합니다.
수동 검수와 자동 평가 결과를 함께 기록할 수 있는 구조가 갖추어져야만 정량 결과와 정성 평가를 동시에 확인할 수 있으며, 이를 통해 평가 신뢰도를 높일 수 있습니다.

실무적으로는 다음 세 가지 점검 질문이 중요합니다.

1. 평가셋이 최신 운영 질의를 반영하고 있는가
2. 정답 문서와 정답 Chuck이 명시적으로 연결되어 있는가
3. 동일 샘플에 대해 자동 점수와 검수 의견을 함께 확인할 수 있는가

### 4.2 검색과 생성의 분리 관리 (Retrieval vs. Generation)

두 번째 학습 사항은 검색 성공과 생성 성공이 서로 다른 품질 문제라는 점입니다.
검색 품질은 정답 문서를 적절히 회수하는 문제입니다.
생성 품질은 회수된 근거를 바탕으로 안정적인 답변을 생성하는 문제입니다.
따라서 Hybrid Search(하이브리드 검색), BM25, Reranker, Top-K와 같은 검색 조합 개선과 환각(Hallucination) 억제, 답변 표현 안정화 등 생성 개선은 분리하여 관리해야 합니다.

실제 운영 환경에서는 검색이 성공하더라도 생성 단계에서 근거 누락, 답변 왜곡, hallucination이 발생할 수 있습니다.
반대로 생성 모델이 우수하더라도 검색이 실패하면 최종 품질은 제한될 수밖에 없습니다.
이에 따라 검색 실패와 생성 실패를 별도로 측정하고 분류하는 체계가 필요합니다.

이 구분은 원인 분석뿐 아니라 조직 간 역할 정의에도 중요합니다.
검색 실패는 데이터, 색인, 메타데이터, 질의 해석 문제와 관련될 가능성이 높고, 생성 실패는 프롬프트, 응답 형식 제약, 모델 응답 안정성과 관련될 가능성이 높습니다.
따라서 동일 지표 하나로 두 문제를 관리할 경우 개선 책임과 실행 우선순위가 불명확해질 수 있습니다.

### 4.3 컨텍스트와 워크플로우 설계 (Context & Workflow)

세 번째 학습 사항은 단순한 컨텍스트 입력량보다 컨텍스트 정제와 워크플로우 설계가 품질에 더 큰 영향을 준다는 점입니다.
많은 문서를 한 번에 입력하는 방식보다, 상위 1~3개 컨텍스트를 선별하여 정제하는 접근이 더 안정적인 성능과 낮은 노이즈를 보일 가능성이 높습니다.

Context Refinement(컨텍스트 정제), Self-Verification(자기 검증), Self-Consistency(자기 일관성)와 같은 후처리 전략은 grounded answer 품질 향상에 직접적인 영향을 미칩니다.
또한 문서 간 오염과 상호 영향도까지 함께 점검해야만 안정적인 품질 운영 체계를 구축할 수 있습니다.

이는 "더 많은 컨텍스트가 항상 더 좋은 품질을 만든다"는 가정이 성립하지 않을 수 있음을 의미합니다.
실제로는 불필요한 컨텍스트가 증가할수록 모델이 부정확한 문장을 참고하거나 상충되는 정보를 혼합할 가능성이 커질 수 있으므로, 입력량 확대보다 입력 품질 관리가 우선되어야 합니다.

## 5. 결과 달성을 위한 실행 과제 (Action Plan)

### 5.1 단기 최우선 과제 (Priority Actions)

단기적으로 우선 수행해야 할 과제는 관리 항목을 채우기 위한 작업이 아니라, 목표 결과를 확보하기 위한 실행 항목으로 이해해야 합니다.
우선 과제는 다음과 같습니다.

1. 학습 및 검증 세트를 확보하고 최신화합니다.
2. 정답지와 Chuck 적합성을 점검합니다.
3. 검색 실패 원인 코드 체계를 정비합니다.
4. 검색 성공 후 생성 실패 케이스를 별도 관리합니다.
5. 운영 반영 전 회귀 케이스와 부작용 점검 기준을 마련합니다.

이 과제들은 선언 수준에서 끝나서는 안 됩니다.
각 과제는 산출물과 완료 기준이 함께 정의되어야 합니다.
실제 프로젝트 관리 항목으로 분해되어야 합니다.
예를 들어 "검증 세트 최신화"는 로그 수집, 질의 유형 분류, 샘플 선정, 정답 작성, Chuck 적합성 검증까지 포함하는 작업으로 이해해야 합니다.

| 실행 과제 (Action) | 만들어야 하는 결과 (Expected Outcome) |
|---|---|
| 검증 세트 최신화 | 최신 운영 질의를 반영한 평가 기준 확보 |
| 정답지 및 Chuck 적합성 점검 | 평가 결과 해석 가능 상태 확보 |
| 검색 실패 코드 체계 정비 | 검색 품질 저하 원인 분해 가능 상태 확보 |
| 생성 실패 케이스 분리 관리 | 생성 품질 개선 우선순위 도출 가능 상태 확보 |
| 회귀 및 배포 기준 마련 | 운영 반영 판단의 일관성 확보 |

### 5.2 목표 결과별 개선 방향 (Improvement Directions by Outcome)

| 목표 결과 (Target Outcome) | 핵심 개선 방향 (Key Direction) |
|---|---|
| 평가 신뢰도 확보 | 정합성 검증, Baseline 재측정, 평가 로그 정비 |
| 검색 원인 분석 가능 | Hybrid Search(하이브리드 검색), BM25, Reranker, Top-K 최적화와 실패 코드 분류 |
| 생성 품질 안정화 | Prompt 개선, hallucination 억제, 출처 생성 흐름 검증 |
| 운영 통제 가능 | 실패 분류, 회귀 추적, 주차별 기록 표준화 |

평가 신뢰도 확보는 전체 개선의 출발점입니다.
검색 원인 분석 가능 상태가 만들어져야 검색 개선 효과를 분명히 볼 수 있습니다.
생성 품질 안정화는 답변 품질의 최종 체감 수준을 좌우합니다.
운영 통제 가능 상태는 이러한 개선 결과가 실제 서비스에 안정적으로 반영되도록 만듭니다.

이 네 가지 목표 결과는 독립적으로 존재하지 않습니다.
평가 기준이 불안정하면 검색 실험의 의미가 약해지고, 검색 품질이 낮으면 생성 품질 개선 효과 역시 제한됩니다.
또한 운영 기준이 없으면 개선 결과가 다음 배포와 회고에 축적되지 못합니다.
따라서 실행 우선순위는 체크리스트 순서가 아니라, 결과 달성 순서에 맞춰 설계되어야 합니다.

### 5.3 단계별 실행 순서 (Execution Sequence)

실행 순서는 `평가 고정 -> 검색 개선 -> 생성 개선 -> 운영 고정`이 바람직합니다.
평가 기준이 먼저 고정되어야 후속 실험 결과를 신뢰할 수 있습니다.
운영 기준이 마지막에 정착되어야 개선 경험이 다음 배포와 후속 과제로 이어질 수 있기 때문입니다.

## 6. 조직 공유 및 확산 포인트 (Sharing & Scale-up)

이번 결과물은 특정 과제의 일회성 보고서로 한정되지 않습니다.
다음 항목은 조직 차원에서 재사용 가능한 공통 자산으로 확장 가능합니다.

- 평가셋 정합성 검증 기준
- 정답 Chuck 적합성 기준
- 검색 실패 코드북
- 생성 실패 분리 관리 기준
- 주간 기록 포맷과 운영 표준

이 자산들은 데이터 담당 조직, 검색/프롬프트/모델 담당 조직, 운영 조직이 공통으로 활용할 수 있습니다.
특히 검색 실패 코드북과 주간 기록 포맷은 단순한 문서가 아니다.
여러 과제가 동일한 언어와 기준으로 품질 상태를 비교할 수 있게 만드는 운영 기준이라는 점에서 의미가 크습니다.
결국 본 프레임워크의 목적은 개별 프로젝트를 지원하는 데 그치지 않습니다.
표준화된 품질관리 방식으로 확장 가능한 공통 방법론을 마련하는 데 있습니다.

## 7. 결론 및 다음 단계 (Conclusion & Next Steps)

이번 정리를 통해 다음 세 가지 결론을 도출하였습니다.

1. 답변 품질 개선의 핵심은 모델 교체보다 구조적 정렬에 있습니다.
2. 평가 기반의 신뢰도를 먼저 확보해야 개선 효과를 정확히 해석할 수 있습니다.
3. 검색과 생성은 분리 개선해야 하며, 운영 추적 체계까지 연결되어야 합니다.

다음 단계는 높음 우선순위 미착수 5건에 우선 착수하여 운영 리스크를 낮추는 것입니다.
동시에 실패 분류 체계와 회귀 기준을 문서화하고, 주차별 기록 체계를 운영 표준으로 정착시켜야 합니다.
향후 성과 판단 또한 단순 완료 건수보다, 아래와 같은 구조적 결과 지표를 기준으로 수행할 필요가 있습니다.

1. 평가 신뢰도 확보 여부
2. 검색 실패와 생성 실패의 분리 측정 정착 여부
3. 회귀 기준과 배포 게이트의 운영 반영 여부
4. 주간 기록 체계의 지속 운영 여부

## 부록 B. 용어 정리 (Appendix B. Terminology)

| 용어 | 설명 |
|---|---|
| 골든셋 | 품질 평가의 기준이 되는 대표 질의-정답 샘플 묶음입니다. 모델, 검색, 생성 개선 효과를 비교할 때 기준선 역할을 합니다. |
| Chuck 적합성 | 원문 문서가 검색 및 생성에 적합한 단위로 잘 분할되고 연결되어 있는 상태를 의미합니다. 정답 문서와 답변 근거를 안정적으로 연결하기 위한 품질 조건입니다. |
| Baseline 정확도 | 개선 전 기준 성능을 의미합니다. 모든 후속 실험은 이 기준 성능과 비교해 해석합니다. |
| Hybrid Search (하이브리드 검색) | 키워드 검색과 벡터 검색을 결합한 방식입니다. 회수율과 정밀도를 동시에 확보하려는 검색 전략입니다. |
| BM25 | 전통적인 키워드 기반 검색 랭킹 알고리즘입니다. 조항형 문서나 명시적 키워드가 중요한 도메인에서 유용합니다. |
| Reranker | 1차 검색 결과를 재정렬하는 후속 랭킹 단계입니다. 정답 문서의 상위 노출 가능성을 높이는 데 사용됩니다. |
| Top-K | 검색 결과 중 상위 몇 개를 컨텍스트로 사용할지 정하는 기준값입니다. 과소 설정 시 근거 누락, 과다 설정 시 노이즈 증가 문제가 발생할 수 있습니다. |
| Hallucination | 실제 근거나 출처 없이 그럴듯한 허위 내용을 생성하는 현상입니다. 운영 신뢰도를 저해하는 대표적인 생성 품질 리스크입니다. |
| Context Refinement (컨텍스트 정제) | 검색된 문서 중 실제 답변에 필요한 부분만 정제하는 과정입니다. 입력 품질을 높여 grounded answer 가능성을 높이는 데 목적이 있습니다. |
| Self-Verification (자기 검증) | 모델이 생성한 답변을 다시 점검하게 하는 후처리 방식입니다. 생성 이후 오류를 줄이는 추가 검증 단계로 활용할 수 있습니다. |
| Self-Consistency (자기 일관성) | 여러 생성 결과 중 더 일관된 답변을 선택하는 전략입니다. 응답 편차가 큰 질문에서 안정성을 높이는 데 효과적입니다. |
| Grounded Answer | 확인 가능한 근거 문서나 출처에 기반해 생성된 답변입니다. 품질개선 프레임워크가 지향하는 최종 답변 상태를 의미합니다. |
