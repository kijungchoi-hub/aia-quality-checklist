## 1. 문서 목적 (Purpose)

본 문서는 `LLM + RAG` 기반 생성형 AI 서비스에서 최종 답변 품질을 체계적으로 개선하기 위한 실무형 기법서입니다.
기존 보고서가 왜 품질 프레임워크가 필요한지를 설명하는 문서라면, 이 문서는 실제 현업 조직이 어떤 순서로 진단하고 무엇을 산출물로 만들어야 하는지 정의합니다.
또한 어떤 기준으로 품질 개선 결과를 판단해야 하는지도 정리한 실행 문서입니다.

본 기법서의 목적은 세 가지입니다.

1. 답변 품질 문제를 `데이터`, `검색`, `생성`, `컨텍스트`, `모델`, `운영`의 상호연결 구조로 해석하는 공통 Framework를 제공합니다.
2. 개선 활동을 단건 실험이 아니라 `진단 (Diagnosis) - 개선 (Improvement) - 검증 (Validation) - 운영화 (Operationalization)` 흐름으로 표준화합니다.
3. 프로젝트마다 반복되는 품질 문제를 재현 가능하고 비교 가능한 방식으로 관리할 수 있도록 공통 산출물과 운영 기준을 정의합니다.

## 2. 왜 Framework가 필요한가 (Why Framework)

`LLM + RAG` 서비스의 최종 답변 품질은 단일 컴포넌트의 성능으로 설명되지 않습니다.
동일한 품질 저하 현상도 실제로는 다음과 같이 서로 다른 원인에서 발생할 수 있습니다.

- 평가셋 정합성 부족으로 개선 효과가 잘못 해석되는 경우
- 검색 실패로 정답 근거가 유입되지 않는 경우
- 검색은 성공했지만 생성 단계에서 hallucination이 발생하는 경우
- 과도한 컨텍스트로 노이즈가 증가하는 경우
- 모델 튜닝 효과와 검색 개선 효과가 혼합되어 원인 분석이 흐려지는 경우
- 운영 로그와 회귀 기준이 없어 개선 결과가 축적되지 않는 경우

즉, 답변 품질 개선은 `복합 시스템 최적화`의 문제입니다.
따라서 개별 기법을 나열하는 방식이 아니라, 원인 분해와 실행 순서를 통제하는 Framework가 필요합니다.

## 3. Framework 구조 (Framework Structure)

본 Framework는 6개 관리 축과 4개 운영 단계로 구성합니다.

### 3.1 관리 축 (Control Towers)

| 관리 축 (Control Tower) | 관리 질문 (Key Question) | 핵심 결과물 (Output) |
|---|---|---|
| 데이터 및 평가 (Data & Evaluation) | 지금 측정하는 결과를 신뢰할 수 있는가 | 평가셋, 정답지, Chuck 적합성 관리표, 평가 로그 |
| 검색 품질 (Retrieval) | 정답 근거가 안정적으로 회수되는가 | 검색 실험표, 실패 코드북, 랭킹 진입 분석 |
| 생성 품질 (Generation) | 근거 기반 답변이 안정적으로 생성되는가 | 생성 실패 케이스북, 프롬프트 정책, 응답 검증표 |
| 컨텍스트 설계 (Context Design) | 필요한 근거만 정확히 전달되는가 | 컨텍스트 정책, 정제 규칙, 워크플로우 비교표 |
| 모델 운영 (Model Operations) | 모델 변경 효과를 분리해서 해석할 수 있는가 | 모델 비교표, 회귀 테스트 결과, 반영 판단서 |
| 운영 관리 (Operations & Governance) | 개선 결과가 운영 표준으로 축적되는가 | 주간 리포트, 배포 게이트, 이슈 트래커 |

### 3.2 운영 단계 (Operating Stages)

| 단계 (Stage) | 목적 (Purpose) | 질문 (Key Question) |
|---|---|---|
| 진단 (Diagnosis) | 현재 품질 저하의 원인을 분해 | 어디에서 문제가 발생하는가 |
| 개선 (Improvement) | 원인에 맞는 개선안을 설계하고 실행 | 무엇을 바꿔야 하는가 |
| 검증 (Validation) | 개선 효과와 부작용을 함께 확인 | 실제로 좋아졌는가 |
| 운영화 (Operationalization) | 재현 가능하게 운영 기준으로 내재화 | 다음 배포와 후속 과제에 남는가 |

### 3.3 성숙도 단계 (Maturity Levels)

| 단계 (Level) | 상태 설명 (Description) | 판단 기준 (Decision Criteria) |
|---|---|---|
| Level 1. 가시화 (Visibility) | 실패 사례는 보이나 구조적 분해가 불충분함 | 로그와 사례는 있으나 분류 기준이 없음 |
| Level 2. 분해 (Decomposition) | 검색/생성/운영 이슈를 분리 추적 가능 | 실패 코드와 케이스북이 존재함 |
| Level 3. 통제 (Control) | 회귀 기준과 배포 게이트로 운영 반영 통제 가능 | 배포 승인 기준과 테스트셋이 운영됨 |
| Level 4. 표준화 (Standardization) | 공통 템플릿과 코드북이 조직 내 재사용됨 | 타 프로젝트 이관이 가능함 |
| Level 5. 최적화 (Optimization) | 지표 기반 우선순위 조정과 지속 개선이 작동함 | 월간/분기 리뷰로 표준이 갱신됨 |

## 4. 적용 원칙 (Principles)

### 4.1 결과 중심 원칙 (Outcome-first)

품질 개선은 `몇 개를 했다`가 아니라 `어떤 결과 상태를 확보했는가`를 기준으로 관리합니다.
예를 들어 `Reranker 교체 완료`는 결과가 아니라 활동입니다.
실제 결과는 `정답 Chuck Top-3 진입률 개선`처럼 표현되어야 합니다.

### 4.2 원인 분리 원칙 (Cause Separation)

검색 실패와 생성 실패를 분리해서 측정합니다.
모델 튜닝 효과와 검색 개선 효과도 분리해서 해석합니다.
평가 기준 변경이 있었으면 이전 결과와 직접 비교하지 않습니다.

### 4.3 산출물 중심 원칙 (Artifact-driven)

모든 개선 활동은 산출물로 남아야 합니다.
산출물이 없는 개선은 재현하기도 어렵고, 다음 프로젝트로 이전하기도 어렵습니다.

### 4.4 운영 반영 원칙 (Operational Readiness)

실험이 좋아졌다고 바로 운영 반영하지 않습니다.
회귀 테스트, 실패 케이스, 부작용, 배포 게이트를 함께 통과해야 운영 반영 후보가 됩니다.

## 5. 실제 적용 절차 (Implementation Flow)

### 5.1 1단계. 진단 (Diagnosis)

목적은 `문제 현상`을 `원인 후보`로 분해하는 것입니다.

#### 수행 항목 (Checklist)

1. 평가셋 최신성 확인
2. 정답지와 Chuck 적합성 확인
3. 검색 실패와 생성 실패 분리 측정
4. 실패 샘플 유형 분류
5. 최근 운영 로그 기반 고빈도 질문군 추출

#### 분석 관점 (Diagnosis Angles)

| 진단 항목 (Diagnosis Item) | 확인 내용 (What to Check) | 대표 증상 (Typical Symptom) |
|---|---|---|
| 평가 불안정 | 정답지 불일치, Chuck 오매핑, 오래된 평가셋 | 실험마다 결과 해석이 흔들림 |
| 검색 실패 | 정답 문서 미회수, 랭킹 하락, 메타데이터 누락 | 관련 근거가 응답에 없음 |
| 생성 실패 | 근거 누락, 답변 왜곡, hallucination | 문장은 자연스럽지만 사실이 틀림 |
| 컨텍스트 실패 | 과다 입력, 상충 문서 혼합, 노이즈 유입 | 답변이 장황하거나 초점이 흐림 |
| 모델 실패 | 특정 질의군에만 취약, 형식 안정성 저하 | 같은 근거로도 답변 편차가 큼 |
| 운영 실패 | 회귀 기준 없음, 이력 누락, 부작용 미추적 | 개선 이후 원복 여부 판단이 어려움 |

#### 필수 산출물 (Required Outputs)

- `evaluation_dataset.csv`
- `answer_key_with_chuck_fit.csv`
- `failure_casebook.md`
- `weekly_quality_log.md`

### 5.2 2단계. 개선 설계 (Improvement Design)

목적은 진단 결과를 `실행 가능한 개선 액션`으로 바꾸는 것입니다.

#### 설계 규칙 (Design Rules)

1. 한 번에 한 축만 크게 바꾼습니다.
2. 변경 항목과 비교 기준을 명시합니다.
3. 기대 효과와 리스크를 함께 적습니다.
4. 실험 성공 기준과 중단 기준을 함께 적습니다.

#### 개선 설계 템플릿 (Design Template)

| 항목 (Field) | 작성 기준 (Guide) |
|---|---|
| 개선 대상 (Target) | 데이터, 검색, 생성, 컨텍스트, 모델, 운영 중 하나 |
| 변경 내용 (Change) | 무엇을 바꾸는지 한 문장으로 명시 |
| 가설 (Hypothesis) | 왜 좋아질 것으로 보는지 |
| 비교 기준 (Baseline) | 무엇과 비교할 것인지 |
| 영향 범위 (Scope) | 어떤 질의군과 어떤 기능에 영향이 있는지 |
| 성공 기준 (Success Metric) | 개선 판정 기준 수치 |
| 부작용 확인 (Risk Check) | 악화될 수 있는 항목 |
| 산출물 (Artifact) | 남길 파일, 로그, 표 |

#### 예시 (Example)

| 항목 | 예시 |
|---|---|
| 개선 대상 | 검색 품질 (Retrieval) |
| 변경 내용 | `BM25 + Dense + Reranker` 조합으로 기본 검색 흐름 변경 |
| 가설 | 정답 Chuck의 Top-3 진입률이 개선될 것이다 |
| 비교 기준 | 기존 Dense only 검색 |
| 성공 기준 | Top-3 진입률 8%p 이상 향상, 오탐률 증가 2%p 이하 |
| 부작용 확인 | 응답 지연 시간 증가, 불필요 문서 유입 여부 |
| 산출물 | 검색 비교표, 실패 코드북 갱신본 |

### 5.3 3단계. 검증 (Validation)

목적은 `좋아진 것처럼 보이는 변화`와 `실제로 좋아진 변화`를 구분하는 데 있습니다.

#### 검증 항목 (Validation Checklist)

1. 동일 평가셋으로 전후 비교
2. 자동 평가와 수동 검수 병행
3. 검색 지표와 생성 지표 분리 확인
4. 주요 실패 케이스 재확인
5. 회귀 테스트 수행
6. 운영 로그 유사 질의군 재시뮬레이션

#### 검증 지표 예시 (Validation Metrics)

| 구분 (Category) | 지표 (Metric) | 의미 (Meaning) |
|---|---|---|
| 검색 (Retrieval) | Top-K Hit Rate | 정답 Chuck 진입 여부 |
| 검색 (Retrieval) | Noise Ratio | 불필요 문서 유입 정도 |
| 생성 (Generation) | Grounded Answer Rate | 근거 기반 답변 비율 |
| 생성 (Generation) | Hallucination Rate | 사실 왜곡 발생 비율 |
| 사용자 관점 (UX) | Manual Review Score | 사람이 느끼는 응답 품질 |
| 운영 관점 (Ops) | Regression Pass Rate | 기존 정상 케이스 유지 여부 |

#### 검증 판단 기준 (Decision Gate)

- `Adopt`: 개선 효과가 명확하고 부작용이 관리 가능함
- `Revise`: 부분 효과는 있으나 추가 조정이 필요함
- `Reject`: 효과가 없거나 부작용이 큼

#### 검증 실패 시 조치 (Failure Handling)

1. 실패 원인을 `평가`, `검색`, `생성`, `운영` 축으로 다시 태깅합니다.
2. 부작용이 전체 지표 악화인지 특정 질의군 편차인지 구분합니다.
3. 재실험 시 기존 Baseline과 비교 기준을 유지합니다.
4. 배포 제외 사유를 주간 리포트와 코드북에 동시에 기록합니다.

### 5.4 4단계. 운영화 (Operationalization)

목적은 실험 결과를 운영 표준으로 내재화하는 데 있습니다.

#### 운영화 항목 (Operational Tasks)

1. 배포 게이트 정의
2. 회귀 테스트셋 등록
3. 실패 코드북 갱신
4. 주간 리포트에 반영
5. 다음 개선 우선순위와 연결

#### 운영 표준 예시 (Operational Standards)

| 운영 항목 (Operational Item) | 기준 (Standard) |
|---|---|
| 배포 승인 | 수동 검수와 자동 평가 모두 기준 통과 |
| 회귀 점검 | 핵심 질의군 100% 재검 |
| 장애 추적 | 실패 유형 코드와 원인 메모 필수 |
| 변경 이력 | 변경 사유, 비교 기준, 결과 수치 기록 |
| 공유 방식 | 주간 보고와 월간 품질 회고에 동시 반영 |

#### 운영화 완료 조건 (Definition of Done)

1. 배포 승인 또는 보류 사유가 문서로 남아 있습니다.
2. 동일 유형 실패에 재사용 가능한 코드와 기준이 있습니다.
3. 회귀 테스트셋이 다음 변경에도 그대로 사용 가능합니다.
4. 다음 리뷰 주기에서 동일 포맷으로 결과를 다시 확인할 수 있습니다.

## 6. 영역별 기법 (Techniques by Area)

### 6.1 데이터 및 평가 (Data & Evaluation)

#### 적용 기법 (Techniques)

- 평가셋 최신화
- 골든셋 재정비
- 정답지-Chuck 적합성 검증
- 수동 검수 + 자동 평가 병행
- 질의 유형군 분리 평가

#### 잘못된 접근 (Anti-pattern)

- 오래된 평가셋으로 새 기능을 판정
- 정답 문서만 있고 정답 Chuck은 없는 상태
- 자동 점수만 보고 실제 답변을 읽지 않음

#### 권장 산출물 (Recommended Artifacts)

- `evaluation_dataset_v2.csv`
- `manual_review_log.xlsx`
- `metric_definition.md`

### 6.2 검색 품질 (Retrieval)

#### 적용 기법

- Hybrid Search
- BM25 조합 비교
- Reranker 교체 실험
- Top-K 및 Weight 최적화
- Query Expansion 검증
- 실패 코드 분류

#### 진단 질문

1. 정답 문서가 아예 안 들어오는가
2. 들어오지만 랭킹이 낮은가
3. 메타데이터 필터가 과도한가
4. 질의 해석이 문서 구조와 안 맞는가

### 6.3 생성 품질 (Generation)

#### 적용 기법

- 프롬프트 구조화
- hallucination 억제 규칙
- 출처 포함 응답 정책
- 생성 실패 케이스북 운영
- 답변 길이와 형식 제약 조정

#### 진단 질문

1. 검색 성공 후에도 오답이 발생하는가
2. 근거는 있는데 답변이 왜곡되는가
3. 응답 형식이 지시를 안정적으로 따르는가
4. 출처를 잘못 연결하는가

### 6.4 컨텍스트 설계 (Context Design)

#### 적용 기법

- Context Refinement
- Top-1/Top-3 컨텍스트 비교
- 문서 간 충돌 감지
- Self-Verification
- Self-Consistency

#### 진단 질문

1. 많이 넣을수록 오히려 품질이 떨어지는가
2. 상충 문서가 동시에 들어오는가
3. 답변에 필요 없는 문서가 자주 섞이는가

### 6.5 모델 운영 (Model Operations)

#### 적용 기법

- SFT 비교
- 선호도 학습 비교
- 모델별 질의군 취약점 분석
- 회귀 테스트셋 기반 판정

#### 진단 질문

1. 같은 검색 결과에서도 모델별 응답 편차가 큰가
2. 특정 질의군만 유난히 약한가
3. 형식 일관성이 모델별로 다른가

### 6.6 운영 관리 (Operations & Governance)

#### 적용 기법

- 주간 품질 리포트
- 실패 코드북
- 배포 게이트
- 품질 회고 템플릿
- 개선 이력 추적표

#### 진단 질문

1. 지난달 실패 케이스를 지금 다시 찾을 수 있는가
2. 왜 배포했는지 또는 왜 보류했는지 설명 가능한가
3. 다음 프로젝트가 그대로 재사용할 수 있는가

## 7. 역할 체계 (Roles & Responsibilities)

| 역할 (Role) | 주요 책임 (Responsibility) |
|---|---|
| 서비스 오너 (Service Owner) | 목표 결과 정의, 우선순위 승인, 운영 반영 판단 |
| 데이터 담당 (Data Owner) | 평가셋, 정답지, 정합성 관리 |
| 검색 담당 (Retrieval Owner) | 검색 실험, 실패 코드, 랭킹 개선 |
| 프롬프트/응답 담당 (Generation Owner) | 응답 정책, 생성 실패 개선, 출처 정책 |
| 모델 담당 (Model Owner) | 튜닝 전략, 모델 비교, 회귀 결과 검토 |
| 운영 담당 (Ops Owner) | 배포 게이트, 주간 기록, 이슈 추적 |

| 배포 승인 | 서비스 오너 | Ops Owner | 전 영역 담당자 |

## 10. 최소 시작 패키지 (Minimum Starter Kit)

Framework를 바로 적용하려면 아래 5개 산출물부터 준비하면 됩니다.

1. 최신 운영 질의를 반영한 평가셋
2. 정답지와 Chuck 적합성 관리표
3. 검색 실패/생성 실패 분리 로그
4. 주간 품질 리포트 템플릿
5. 배포 게이트 체크리스트

이 다섯 가지가 준비되면, 고도화 이전에도 최소한의 품질 개선 루프를 운영할 수 있습니다.

## 12. 결론 (Conclusion)

Answer Quality Framework의 핵심은 기법의 개수에 있지 않습니다.
실제 핵심은 `품질 저하를 어디서 어떻게 분해할 것인가`, `개선 효과를 무엇으로 증명할 것인가`, `좋아진 결과를 어떻게 운영 표준으로 남길 것인가`에 있습니다.

따라서 본 기법서는 체크리스트를 확장한 참고 문서가 아니라, 생성형 AI 서비스 구축 단계에서 품질 개선 활동을 설계하고 실행하고 운영화하기 위한 실무 표준으로 사용해야 합니다.
이후 각 프로젝트는 본 문서를 기반으로 도메인 특화 지표와 질의군만 추가하면 됩니다.
