## 1. 보고 개요 (Overview)

본 보고서는 `answer` 품질 개선 활동을 주요 학습 사항, 실행 과제, 공유 및 확산 관점에서 재정리한 문서다.
목적은 개선 과제를 나열하는 데 있지 않다.
데이터, 검색, 생성, 운영의 연결 구조 안에서 어떤 목표 상태가 필요한지 정리하는 데 있다.
또한 무엇을 먼저 실행해야 하는지도 재사용 가능한 형태로 정리하는 데 있다.

핵심 메시지는 명확하다.
답변 품질 개선의 핵심은 모델 교체 자체가 아니라, 신뢰 가능한 평가 기반을 갖추고 검색과 생성을 분리 개선하며 이를 운영 가능한 기록 체계로 축적하는 데 있다.

즉, 본 보고서는 단순한 현황 보고서가 아니라 "무엇을 배웠고, 무엇을 당장 실행해야 하며, 무엇을 조직에 남길 것인가"를 정리한 실행형 보고서라고 볼 수 있다.
이 점에서 장표용 문구를 그대로 옮긴 문서가 아니라, 발표 내용을 실무 문서로 확장한 버전이라는 성격을 가진다.

## 2. 추진 배경 (Background)

기존 답변 품질 개선 활동은 데이터, 검색, 생성, 운영 영역에 분산되어 관리되어 왔다.
이 과정에서 동일 문제를 다른 이름으로 반복 관리하거나, 개선 결과를 운영 반영 판단과 연결하기 어려운 한계가 있었다.
또한 회귀 추적과 배포 기준이 충분히 정리되지 않아, 실험 결과가 실제 운영 품질 개선으로 이어지는 경로가 명확하지 않았다.

특히 기존 방식에서는 "모델 성능 저하", "검색 품질 부족", "근거 누락", "운영 기준 미비" 같은 현상이 각각 따로 관리되기 쉬웠다.
하지만 실제 사용자 경험에서는 이 문제가 분리되어 나타나지 않는다.
예를 들어 검색 회수율이 낮아도 답변 품질이 낮아지고, 검색은 성공했지만 생성 단계에서 근거가 왜곡돼도 최종 품질은 낮아진다.
이 때문에 문제를 기능 단위가 아니라 서비스 흐름 단위로 묶어 보는 기준이 필요했다.

따라서 이번 정리는 실행 가능한 공통 프레임을 만드는 데 초점을 두었다.
기존 관리 항목을 목표 결과 중심으로 재구성하고, 주요 학습 사항, 실행 과제, 공유 및 확산 관점으로 재정리함으로써 개별 실험 중심의 접근을 운영 가능한 품질 관리 체계로 전환하고자 했다.
다시 말해, 개선 활동을 "실험의 누적"이 아니라 "품질 운영 체계의 구축"으로 해석하는 관점 전환이 이번 정리의 출발점이다.

## 3. 목표 결과 중심 현황 (Outcome Status)

이 절에서 중요한 것은 관리 대상의 수량이 아니라, 어떤 결과 상태를 확보해야 하는가이다.
따라서 현황은 진행상태보다 목표 상태와 개선점 중심으로 해석해야 한다.

| 목표 결과 (Target Outcome) | 개선점 (Improvement Point) | 개선방향 (Improvement Direction) |
|---|---|---|
| 신뢰 가능한 평가 기반 확보 | 평가셋 정합성과 기준 통일이 완전하지 않음 | 검증 세트 최신화, 정답지/Chuck 적합성 보강 |
| 검색 품질의 원인 분해 가능 | 검색 실패 유형이 체계적으로 누적되지 않음 | 실패 코드북 정비, 검색 조합 비교 실험 |
| 생성 품질의 안정적 통제 | 검색 성공 후 생성 실패가 별도 관리되지 않음 | 생성 실패 케이스북 운영, 출처 흐름 검증 |
| 운영 반영 판단의 일관성 확보 | 회귀 기준과 배포 게이트가 충분히 정착되지 않음 | 회귀 테스트셋, 배포 게이트 기준 운영 |

현재 시점에서 중요한 것은 새로운 과제를 늘리는 것이 아니라, 남아 있는 핵심 개선점을 해소하고 실행 순서를 정렬하는 일이다.

이 표가 의미하는 바는 분명하다.
관리해야 할 것은 목록 자체가 아니라 결과 상태다.
따라서 향후 보고와 운영 회의에서는 전체 건수보다 결과 상태 확보 여부, 개선점 해소 정도, 개선방향의 실행 여부를 우선 추적하는 것이 실무적으로 더 유의미하다.

## 4. 주요 학습 사항 (Key Learnings)

### 4.1 데이터와 평가 기준 (Data & Evaluation)

가장 먼저 확인된 학습 포인트는 데이터와 평가 기준의 정합성이 모든 품질 개선의 전제 조건이라는 점이다.
골든셋, 정답지, Chuck 적합성이 확보되지 않으면 Baseline 정확도와 실패 사례를 해석할 수 없고, 검색 개선인지 모델 개선인지조차 정확히 구분하기 어렵다.

또한 정답률, 오답률, 소답률, 과답률, 검색 매칭률 등 주요 지표 기준을 통일해야 한다.
수동 검수와 자동 평가 결과를 함께 기록하는 체계가 있어야 정량과 정성 관찰을 함께 보면서 평가 신뢰도를 높일 수 있다.

실무적으로는 세 가지를 먼저 점검해야 한다.
"평가셋이 최신 운영 질의를 반영하고 있는가"를 확인해야 한다.
"정답 문서와 정답 Chuck이 명시적으로 연결되어 있는가"를 확인해야 한다.
"동일 샘플에 대해 자동 점수와 검수 의견을 함께 볼 수 있는가"도 확인해야 한다.
이 세 항목이 충족되지 않으면 수치가 좋아져도 실제 개선인지, 데이터 편향인지, 평가 방식 변경 효과인지 구분하기 어렵다.

따라서 데이터 및 평가 기준 영역에서는 단순히 데이터 양을 늘리는 것보다 평가 신뢰도를 높이는 구조를 먼저 갖추는 것이 중요하다.
이는 이후 검색 개선과 생성 개선의 효과를 해석하는 공통 기반이 되기 때문이다.

### 4.2 검색과 생성의 분리 관리 (Retrieval vs. Generation)

두 번째 학습 포인트는 검색 성공과 생성 성공이 별개의 문제라는 점이다.
검색 품질은 정답 문서를 회수하는 문제이고, 생성 품질은 그 근거를 바탕으로 안정적인 답변을 만드는 문제다.
따라서 Hybrid Search, BM25, Reranker, Top-K 같은 검색 조합 개선과, hallucination 억제 및 답변 표현 안정화 같은 생성 개선은 분리해서 봐야 한다.

실제 운영에서는 검색이 성공해도 생성 단계에서 근거 누락, 답변 왜곡, hallucination이 발생할 수 있다.
반대로 생성 모델이 충분히 좋아도 검색이 실패하면 품질 개선 효과는 제한적이다.
이 때문에 검색 실패와 생성 실패를 분리 측정하고 분류하는 체계가 필요하다.

이 구분은 원인 분석뿐 아니라 책임 분배 측면에서도 중요하다.
검색 실패는 데이터/색인/메타데이터/질의 해석 문제일 가능성이 높고, 생성 실패는 프롬프트, 답변 형식 제약, 모델 응답 안정성 문제일 가능성이 높다.
두 영역을 한 지표로만 관리하면 어느 조직이 무엇을 개선해야 하는지가 모호해진다.

따라서 검색과 생성은 같은 품질 목표에 기여하지만, 개선 활동과 실험 설계는 별도 체계로 운영해야 한다.
이 구조가 갖춰져야 검색 개선 효과와 생성 개선 효과를 분리 해석할 수 있고, 실제 배포 기준도 더 명확하게 설정할 수 있다.

### 4.3 컨텍스트와 워크플로우 (Context & Workflow)

세 번째 학습 포인트는 컨텍스트 입력량보다 컨텍스트 정제와 워크플로우 설계가 더 중요하다는 점이다.
많은 컨텍스트를 무조건 넣는 방식보다는, 상위 1~3개 컨텍스트를 정제해 사용하는 접근이 더 나은 성능과 낮은 노이즈를 보일 가능성이 높다.

Context Refinement, Self-Verification, Self-Consistency 같은 후처리 전략은 단순한 프롬프트 조정보다 더 직접적으로 grounded answer 품질에 영향을 미친다.
또한 문서 간 오염과 상호 영향도까지 점검해야 안정적인 품질 개선이 가능하다.

이는 "더 많이 넣으면 더 정확하다"는 직관이 항상 맞지 않다는 뜻이기도 하다.
실제로는 불필요한 컨텍스트가 많을수록 모델이 엉뚱한 문장을 참고하거나 상충되는 정보를 섞어 답변할 가능성이 높아질 수 있다.
따라서 컨텍스트 전략은 입력량 경쟁이 아니라, 필요한 근거를 얼마나 명확하게 추려 전달하느냐의 문제로 이해해야 한다.

## 5. 실행 과제 (Action Plan)

### 5.1 단기 최우선 실행 과제 (Priority Actions)

단기적으로 가장 먼저 수행해야 할 과제는 관리 항목을 채우기 위한 작업이 아니라, 목표 결과를 확보하기 위한 실행 항목으로 이해해야 한다.
우선 과제는 다음과 같다.

1. 학습 및 검증 세트를 확보하고 최신화한다.
2. 정답지와 Chuck 적합성을 점검한다.
3. 검색 실패 원인 코드 체계를 정비한다.
4. 검색 성공 후 생성 실패 케이스를 별도 관리한다.
5. 운영 반영 전 회귀 케이스와 부작용 점검 기준을 마련한다.

이 과제들은 모두 개념 논의가 아니라 실행 단위로 분해해 관리해야 하며, 각 항목마다 산출물과 완료 기준이 함께 정의되어야 한다.

예를 들어 "검증 세트 최신화"는 단순 문장 하나로 끝나는 과제가 아니라, 운영 로그 수집, 질의 유형 분류, 샘플 선정, 정답 작성, Chuck 적합성 검증까지 포함하는 작업이다.
마찬가지로 "검색 실패 코드 체계화"도 코드 목록을 만드는 것에서 끝나는 것이 아니라, 실제 실패 로그에 태깅하고 집계하며, 원인별 개선 우선순위를 정하는 단계까지 이어져야 한다.

따라서 실행 과제는 모두 "무엇을 할 것인가"뿐 아니라 "어떤 결과 상태를 만들 것인가", "어떤 산출물이 남는가", "완료로 볼 수 있는 기준은 무엇인가"까지 함께 설계해야 한다.
그래야 보고서의 액션 항목이 회의 메모 수준이 아니라 실제 프로젝트 관리 항목으로 작동한다.

### 5.2 목표 결과별 실행 방향 (Improvement Directions by Outcome)

| 목표 결과 (Target Outcome) | 핵심 실행 방향 (Key Direction) |
|---|---|
| 평가 신뢰도 확보 | 정합성 검증, Baseline 재측정, 평가 로그 정비 |
| 검색 원인 분석 가능 | Hybrid Search, BM25, Reranker, Top-K 최적화와 실패 코드 분류 |
| 생성 품질 안정화 | Prompt 개선, hallucination 억제, 출처 생성 흐름 검증 |
| 운영 통제 가능 | 실패 분류, 회귀 추적, 주차별 기록 표준화 |

데이터 축은 평가 기반의 신뢰도 확보를 담당한다.
검색 축은 정답 Chuck 진입률 개선과 실패 원인 분해를 담당한다.
생성 축은 grounded answer 품질을 높이는 역할을 맡는다.
운영 축은 개선 결과를 배포와 회고에 연결하는 관리 체계를 책임진다.

이 네 축은 독립 과제가 아니라 서로 연결된 구조다.
데이터 축이 약하면 검색 품질 실험이 흔들리고, 검색 축이 약하면 생성 축의 개선 효과가 제한되며, 운영 축이 약하면 개선 결과가 다음 배포와 회고에 남지 않는다.
따라서 실행 우선순위를 잡을 때도 특정 축 하나만 강화하는 방식보다는, 최소한의 연결 고리가 유지되도록 균형 있게 설계하는 것이 중요하다.

## 6. 공유 및 확산 (Sharing & Scale-up)

이번 결과물은 단일 과제 보고서로 끝나는 문서가 아니다.
다음과 같은 항목은 조직 차원에서 재사용 가능한 공통 자산으로 확장할 수 있다.

- 평가셋 정합성 검증 기준
- 정답 Chuck 적합성 기준
- 검색 실패 코드북
- 생성 실패 케이스 분리 관리 기준
- 주간 기록 포맷과 운영 표준

이 기준들은 데이터 담당 조직, 검색/프롬프트/모델 담당 조직, 운영 조직 모두가 공통으로 활용할 수 있다.
따라서 이번 정리는 특정 프로젝트의 개선 보고를 넘어 조직 내 공통 품질 프레임으로 확장 가능한 성격을 가진다.

특히 공유 가능한 자산은 "문서"보다 "기준"에 가깝다.
예를 들어 검색 실패 코드북은 특정 프로젝트의 로그 분류표가 아니다.
이후 다른 과제에서도 같은 방식으로 실패를 비교할 수 있게 만드는 공통 언어다.
주간 기록 포맷 역시 단순한 템플릿이 아니다.
서로 다른 프로젝트의 진행 상황을 같은 관점에서 비교하게 만드는 운영 기준으로 이해해야 한다.

## 7. 결론 및 다음 단계 (Conclusion & Next Steps)

이번 정리에서 확인한 결론은 다음과 같다.

1. 답변 품질 개선의 핵심은 모델 교체보다 구조적 정렬에 있다.
2. 평가 기반의 신뢰도를 먼저 확보해야 개선 효과를 정확히 해석할 수 있다.
3. 검색과 생성은 분리 개선해야 하며, 운영 추적 체계까지 연결되어야 한다.

다음 단계는 명확하다.
높음 우선순위 미착수 5건부터 착수해 운영 리스크를 낮추고, 실패 분류 체계와 회귀 기준을 문서화하며, 주차별 기록 체계를 운영 표준으로 정착시켜야 한다.

이후 성과 판단 역시 단순 완료 건수보다, "평가 신뢰도 확보 여부", "검색/생성 분리 측정 정착 여부", "회귀 기준의 운영 반영 여부" 같은 구조적 지표로 보는 것이 바람직하다.
그래야 이번 활동이 일회성 개선이 아니라 반복 가능한 운영 체계로 자리 잡았는지 판단할 수 있다.

## 부록 B. 실제 프로젝트 운영 프레임워크 (Appendix B. Operating Framework)

### 프레임워크 이름 (Framework Name)

- Answer Quality Improvement Framework

### 프레임워크 구조 (Framework Structure)

```text
1. Data Readiness
   - 평가셋 최신화
   - 골든셋/정답지/Chuck 적합성

2. Retrieval Quality
   - 검색 조합 비교
   - Top-K / Weight / Reranker 최적화
   - 실패 원인 코드화

3. Generation Quality
   - Prompt 개선
   - Hallucination 억제
   - 근거/출처 출력

4. Release Control
   - 회귀 테스트
   - 부작용 점검
   - 배포 게이트

5. 운영 학습 체계
   - 주간 기록 표준화
   - 실패 케이스 축적
   - 타 영역 확산 검토
```

### 운영 원칙 (Operating Principles)

1. 검색 실패와 생성 실패는 항상 분리 관리한다.
2. 모델 변경 전에는 데이터셋과 검색 기준을 먼저 고정한다.
3. 모든 개선은 동일 평가셋으로 전후 비교한다.
4. 배포 전에는 회귀 테스트 결과가 있어야 한다.
5. 주간 기록은 결과뿐 아니라 실패와 차주 계획까지 포함한다.

### 단계별 산출물 (Deliverables by Stage)

| 단계 (Stage) | 핵심 산출물 (Key Deliverables) |
|---|---|
| Data Readiness | 평가셋, 정합성 점검표 |
| Retrieval Quality | 검색 조합 비교표, 실패 코드북 |
| Generation Quality | 프롬프트 버전표, 출처 포맷 정의서 |
| Release Control | 회귀 테스트셋, 배포 점검표 |
| 운영 학습 체계 | 주간 보고서, 실패 케이스북, 확산 검증표 |

## 부록 E. 용어 정리 (Appendix E. Terminology)

| 용어 | 설명 |
|---|---|
| 골든셋 | 품질 평가의 기준이 되는 대표 질의-정답 샘플 묶음이다. 모델이나 검색 품질 비교 시 기준선 역할을 한다. |
| Chuck 적합성 | 원문 문서가 검색 및 생성 단계에서 다루기 쉬운 단위로 적절하게 분할되고 연결되어 있는 상태를 의미한다. 검색과 생성이 동일 기준으로 근거를 활용할 수 있게 만드는 품질 조건이다. |
| Baseline 정확도 | 개선 작업 전 기준 성능을 의미한다. 이후 실험 효과를 비교하는 출발점이 된다. |
| Hybrid Search | 키워드 기반 검색과 벡터 기반 검색을 결합해 회수율과 정밀도를 함께 높이려는 검색 방식이다. 질의 표현 다양성과 문서 의미 유사성을 동시에 반영할 때 주로 사용한다. |
| BM25 | 전통적인 키워드 기반 검색 랭킹 알고리즘으로, 질의어와 문서 간 일치도를 점수화하는 데 사용된다. 약관 조항이나 정책 키워드처럼 명시적 표현이 중요한 질의에서 특히 유용하다. |
| Reranker | 1차 검색 결과를 다시 재정렬해 더 관련성 높은 문서를 상단에 배치하는 후속 랭킹 단계다. 초기 회수 이후 정답 문서의 상위 진입률을 높이는 데 활용한다. |
| Top-K | 검색 결과 중 상위 몇 개 문서를 컨텍스트로 사용할지 정하는 기준값이다. 너무 작으면 근거가 누락되고, 너무 크면 노이즈가 증가할 수 있다. |
| Hallucination | 모델이 실제 근거나 출처 없이 그럴듯하지만 사실과 다른 내용을 생성하는 현상을 말한다. 생성형 AI 서비스에서 신뢰도 저하를 일으키는 대표 위험 요소다. |
| Context Refinement | 검색된 문서 중 실제 답변에 필요한 부분만 선별·정제하는 과정이다. 불필요한 문장과 상충 정보를 줄여 grounded answer 품질을 높이는 데 목적이 있다. |
| Self-Verification | 모델이 생성한 답변을 다시 점검하게 해 근거 누락이나 오류를 줄이려는 후처리 방식이다. 답변 생성 이후 품질을 한 번 더 검증하는 통제 장치로 활용할 수 있다. |
| Self-Consistency | 여러 생성 결과를 비교해 더 일관된 답변을 선택하는 전략이다. 응답 편차가 큰 질의군에서 안정성을 높이는 데 효과적이다. |
| Grounded Answer | 검색된 근거 문서나 확인 가능한 출처에 기반해 생성된 답변을 의미한다. 최종 답변 품질 관리에서 가장 중요한 목표 상태 중 하나다. |
