## 슬라이드 2. 추진 배경

### 슬라이드 목적

- 이 발표가 왜 필요한지, 기존 방식의 한계가 무엇인지 설명한다.

### 슬라이드 제목

- 왜 체크리스트 기반 재정리가 필요했는가

### 핵심 메시지

- 답변 품질 이슈가 개별 실험 단위로 흩어져 있어 구조적 관리가 어려웠다.
- 품질 문제를 모델 성능 하나로 설명하기 어려웠다.
- 실험 결과를 운영과 연결할 수 있는 공통 정리 기준이 필요했다.

### 본문 bullet

- `answer` 품질 개선 과제가 데이터, 검색, 프롬프트, 모델, 운영에 분산되어 있었음
- 동일 문제를 다른 표현으로 반복 관리하는 비효율이 있었음
- 실험은 있었지만 운영 반영 판단 기준과 회귀 추적 기준이 충분히 정리되지 않았음
- 따라서 실행 가능한 체크리스트와 LDS 보고 관점을 결합한 관리 틀이 필요했음

### 시각화 제안

- `Before / After` 비교 도식
- Before: 개별 실험 메모, 산발적 개선
- After: 체크리스트 기반 통합 관리, LDS 보고 체계

### 발표 멘트

기존에는 답변 품질 개선 과제가 개별 실험 중심으로 흩어져 있었습니다. 그러다 보니 같은 문제를 다른 이름으로 반복 관리하거나, 개선 결과를 운영 판단으로 연결하기 어려운 한계가 있었습니다. 이번 정리는 이런 산발적인 개선 활동을 체크리스트와 LDS 구조로 묶어서, 설명 가능하고 재사용 가능한 형태로 바꾸는 작업입니다.

### 연결 문장

- 그 기준 위에서 현재 전체 현황을 먼저 보겠습니다.

### 슬라이드 목적

### 슬라이드 제목

- 체크리스트 전체 현황

### 핵심 수치

- 진행중: 25건
- 미착수: 9건
- 높음 우선순위: 24건
- 중간 우선순위: 10건
- 높음 우선순위 미착수: 5건

### 본문 bullet

- 전체적으로 핵심 개선 축은 이미 착수된 상태
- 다만 운영 리스크와 직결되는 높음 우선순위 미착수 항목이 남아 있음
- 따라서 지금 필요한 것은 신규 과제 발굴보다 핵심 공백 해소와 실행 순서 정렬임

### 표 제안

| 구분 | 건수 | 해석 |
|---|---:|---|
| 전체 항목 | 34 | 전체 관리 대상 |
| 진행중 | 25 | 핵심 축은 이미 착수 |
| 미착수 | 9 | 남은 공백 존재 |
| 높음 우선순위 미착수 | 5 | 단기 최우선 실행 대상 |

### 차트 제안

- 도넛차트 1: 진행중 vs 미착수
- 막대차트 1: 높음 / 중간 우선순위

### 발표 멘트

현재 전체 34개 항목 중 25개는 진행중입니다. 즉, 큰 방향성은 이미 잡혀 있습니다. 다만 높음 우선순위인데 아직 착수되지 않은 항목이 5건 남아 있어서, 지금 시점의 핵심은 새로운 주제를 넓히는 것이 아니라 이 공백을 메우는 데 있습니다.

### 연결 문장

## 슬라이드 4. Learn 1 - 데이터와 평가 기준

### 슬라이드 목적

- 품질 개선의 출발점이 데이터 정합성과 평가 신뢰도라는 점을 강조한다.

### 슬라이드 제목

- Learn 1. 데이터와 평가 기준이 흔들리면 결과 해석도 흔들린다

### 핵심 메시지

- 평가셋 정합성은 모든 품질 개선의 전제 조건이다.
- 골든셋, 정답지, Chunk ID가 맞지 않으면 실험 결과를 신뢰할 수 없다.

### 본문 bullet

- 학습 및 검증 세트 확보 여부가 아직 핵심 과제로 남아 있음
- 골든셋, 정답지, Chunk ID 정합성 검증이 높음 우선순위로 관리되고 있음
- Baseline 정확도와 실패 사례 재검이 필요함
- 수동 검수와 자동 평가 결과를 함께 기록해야 함
- 평가 지표는 정답률, 오답률, 소답률, 과답률, 검색 매칭률로 통일할 필요가 있음

### 시각화 제안

- `평가 신뢰도 피라미드`
- 하단: 데이터 정합성
- 중단: 평가 기준 통일
- 상단: 실험 결과 해석

### 발표 멘트

이번에 가장 먼저 확인한 점은 데이터와 평가 기준이 흔들리면 그 위에 쌓이는 모든 실험 결과도 불안정해진다는 점입니다. 특히 골든셋, 정답지, Chunk ID가 정합하지 않으면 모델이 좋아졌는지, 검색이 좋아졌는지 자체를 정확히 판단하기 어렵습니다. 그래서 품질 개선의 첫 출발점은 모델이 아니라 평가 기반의 신뢰도 확보입니다.

### 연결 문장

- 다음은 검색과 생성을 왜 분리해서 봐야 하는지 말씀드리겠습니다.

## 슬라이드 5. Learn 2 - 검색과 생성은 다른 문제다

### 슬라이드 목적

- 검색 성공과 생성 성공을 분리하지 않으면 품질 원인 분석이 왜곡된다는 점을 설명한다.

### 슬라이드 제목

- Learn 2. 검색 성공과 생성 성공은 별도로 관리해야 한다

### 핵심 메시지

- 검색은 정답 문서를 잘 가져오는 문제이고, 생성은 가져온 근거로 답을 안정적으로 만드는 문제다.
- 두 단계를 혼합하면 개선 원인을 잘못 해석하게 된다.

### 본문 bullet

- Hybrid Search를 기본 검색 방식 후보로 검증 중
- Lexical, Vector, BM25, Reranker 조합 비교가 핵심 과제
- Top-K, Weight, 리랭킹 조정이 정답 Chunk 진입률에 직접 영향
- 검색 성공 후에도 hallucination, 근거 누락, 표현 왜곡이 별도 발생
- 따라서 검색 실패와 생성 실패를 구분하는 관리 체계가 필요

### 도식 제안

- 2단계 파이프라인 도식
- 1단계: Query -> Retrieval -> Relevant Chunk 진입
- 2단계: Context -> Generation -> Grounded Answer
- 단계별 실패 예시 표시

### 발표 멘트

중요한 점은 검색 성공과 생성 성공이 같은 문제가 아니라는 것입니다. 정답 문서를 잘 찾았더라도 생성 단계에서 hallucination이 발생하거나, 근거는 맞는데 답변 표현이 왜곡될 수 있습니다. 그래서 검색 성능 개선과 프롬프트 개선, 모델 튜닝 효과를 한꺼번에 보면 원인 분석이 틀어질 수 있습니다. 이 둘은 반드시 분리해서 관리해야 합니다.

### 연결 문장

- 그 다음으로는 컨텍스트와 워크플로우가 왜 중요했는지 보겠습니다.

## 슬라이드 6. Learn 3 - 컨텍스트와 워크플로우가 품질을 만든다

### 슬라이드 목적

- 검색 결과를 어떻게 가공하고 어떤 후처리 흐름을 적용하느냐가 품질에 미치는 영향을 설명한다.

### 슬라이드 제목

- Learn 3. 많이 넣는 것보다 정확하게 넣는 것이 중요하다

### 핵심 메시지

- 컨텍스트 수량보다 컨텍스트 정제와 워크플로우 설계가 더 중요하다.

### 본문 bullet

- Context Refinement 적용 여부가 핵심 검토 항목
- 상위 1~3개 컨텍스트만 사용하는 방식의 성능 검증 필요
- Self-Verification, Self-Consistency 같은 후처리 전략의 실효성 비교 필요
- 문서 간 오염과 상호 영향도도 별도 점검 대상
- 분류형 Workflow가 필요한 영역은 따로 식별해야 함

### 시각화 제안

- `많은 컨텍스트 입력` vs `정제된 컨텍스트 입력` 비교 그림
- 우측에 품질, 지연시간, 노이즈 3개 축 비교

### 발표 멘트

이번 정리에서 확인한 또 하나의 포인트는 많이 넣는 것이 항상 좋은 결과로 이어지지 않는다는 점입니다. 오히려 상위 컨텍스트를 정제해서 넣고, 필요하면 Self-Verification 같은 후처리를 결합하는 방식이 더 중요합니다. 결국 답변 품질은 검색 결과 자체뿐 아니라 그것을 어떻게 구성하고 흘려보내느냐에 의해 크게 좌우됩니다.

### 연결 문장

- Learn에서 확인한 내용을 바탕으로, 이제 실제 실행 과제를 말씀드리겠습니다.

## 슬라이드 7. Do 1 - 즉시 실행 과제

### 슬라이드 목적

- 지금 당장 우선순위를 두고 실행해야 하는 핵심 과제를 명확히 제시한다.

### 슬라이드 제목

- Do 1. 단기 최우선 실행 과제

### 우선 과제

1. 학습 및 검증 세트 확보 및 최신화
2. 검색 실패 원인 코드 체계화
3. 검색 성공 후 생성 실패 케이스 분리 관리
4. 모델 튜닝 효과와 검색 개선 효과 분리 해석
5. 운영 반영 전 회귀 케이스와 부작용 점검

### 본문 bullet

- 위 5개는 높음 우선순위 미착수 또는 운영 리스크와 직결되는 항목
- 이 과제를 선행해야 이후 실험 결과의 설명력이 확보됨
- 신규 실험 확대보다 현재 공백을 메우는 것이 우선

### 표 제안

| 과제 | 목적 | 기대 효과 |
|---|---|---|
| 학습/검증 세트 확보 | 평가 신뢰도 확보 | 비교 실험 기반 안정화 |
| 실패 원인 코드화 | 분석 정밀도 향상 | 액션 연결 속도 증가 |
| 생성 실패 분리 관리 | 원인 분리 | 생성 개선 효율 상승 |
| 효과 분리 해석 | 오판 방지 | 튜닝 투자 우선순위 명확화 |
| 회귀 점검 | 운영 안정성 확보 | 배포 리스크 감소 |

### 발표 멘트

단기적으로 가장 중요한 실행 과제는 이 다섯 가지입니다. 공통점은 모두 결과 해석의 신뢰도와 운영 반영 안정성에 직접 연결된다는 점입니다. 다시 말해, 성능을 더 올리는 실험보다 먼저 이 기반을 메워야 이후 개선이 실제 성과로 이어집니다.

### 연결 문장

- 이 과제를 포함해 전체 실행 방향은 네 개 축으로 정리할 수 있습니다.

## 슬라이드 8. Do 2 - 영역별 실행 방향

### 슬라이드 목적

- 실제 실행 체계를 영역별로 정돈해 보여준다.

### 슬라이드 제목

- Do 2. 실행은 데이터, 검색, 생성, 운영의 네 축으로 정리한다

### 실행 축

#### 데이터

- 골든셋 정합성 재검
- Baseline 재측정
- 수동/자동 평가 결과 동시 기록

#### 검색

- Hybrid Search 검증
- BM25, Reranker, Top-K, Weight 조합 비교
- Query Expansion 효과 검증

#### 생성

- 답변 생성 프롬프트 개선
- hallucination 억제 조건 강화
- 근거와 출처 생성 흐름 검증

#### 운영

- 실패 유형 분류
- 회귀 케이스 추적
- 주차별 실험 로그 포맷 표준화

### 시각화 제안

- 4분면 구조 또는 4개 컬럼 구조
- 각 축 아래 `핵심 과제 3개`씩 배치

### 발표 멘트

전체 실행 방향은 데이터, 검색, 생성, 운영의 네 축으로 단순화했습니다. 데이터는 신뢰도 기반, 검색은 정답 회수율, 생성은 grounded answer 안정화, 운영은 재현성과 회귀 통제로 이해하시면 됩니다. 이 네 축이 맞물려야 실제 운영 가능한 품질 개선이 됩니다.

### 연결 문장

- 마지막으로 이 내용을 조직 차원에서 어떻게 공유하고 확산할지 말씀드리겠습니다.

### 슬라이드 제목

### 다음 단계

## 부록 1. 발표자 참고 메모

### 강조해야 할 포인트

- 모델 성능만으로 설명하지 않는다는 점
- 검색과 생성의 분리 관리 필요성
- 평가 정합성 확보가 최우선이라는 점
- 운영 반영 전 회귀 관리가 반드시 필요하다는 점

### 질문 예상 포인트

1. 왜 모델 튜닝보다 데이터/검색을 먼저 보느냐
2. 검색 성공과 생성 실패를 어떻게 구분할 것이냐
3. 높음 우선순위 미착수 5건 중 무엇부터 시작할 것이냐
4. 이 프레임을 다른 과제에도 동일하게 쓸 수 있느냐

### 답변 가이드

- 모델 튜닝은 중요하지만, 평가 기반과 검색 품질이 불안정하면 개선 효과를 정확히 측정하기 어렵다고 답변
- 검색과 생성은 실패 유형 정의와 로그 구조를 분리해 관리할 계획이라고 답변
- 우선순위는 평가 신뢰도와 운영 리스크 기준으로 정했다고 답변
- 체크리스트 구조는 `summary`나 다른 RAG 과제에도 확장 가능하다고 답변
