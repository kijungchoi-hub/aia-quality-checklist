# 요약 품질 체크리스트

기준 데이터: `data/quality_checklist_raw_summary.csv`

정리 기준:
- 요약 유형별 구분은 최소화합니다.
- 주차별 반복 표현은 제거합니다.
- 동일한 목적의 개선 활동은 통합합니다.
- 실행 가능한 체크리스트와 개선 활동만 남깁니다.

## 최종 체크리스트

### 1. 데이터 및 평가 기준

- [ ] 학습 및 검증 데이터를 확보하고 최신 상태로 유지합니다.
- [ ] 요약 정답 기준과 판정 기준을 명확히 정리합니다.
- [ ] RAGAS, LLM Judge, 수동 검수 기준을 함께 적용합니다.
- [ ] 오답, 과답, 누락 데이터를 기준으로 평가 결과를 누적 관리합니다.
- [ ] 민원/개선 데이터를 추가 반영하고 재평가합니다.
- [ ] 요약 유형별 베이스라인을 동일 형식으로 기록합니다.

### 2. 프롬프트 품질 개선

- [ ] 현재 프롬프트 구조의 정합성을 점검합니다.
- [ ] 프롬프트 개선 포인트를 유형별로 정리합니다.
- [ ] 요약 기준 강화와 제약 조건을 반영합니다.
- [ ] Few-shot, 규칙, 키워드 추출 방식을 보강합니다.
- [ ] 대/중/소 분류와 요약 생성 프롬프트의 역할을 분리해 검토합니다.
- [ ] 과답, 누락, 왜곡을 줄이기 위한 제어 문구를 정교화합니다.

### 3. 오류 분석 및 개선 루프

- [ ] 오류 유형 분석 체계를 고정합니다.
- [ ] 오류 케이스를 정답 가능, 정답 변경 필요, false 데이터 등으로 세분화합니다.
- [ ] 오류 케이스 분석 결과를 프롬프트에 다시 반영합니다.
- [ ] 유형별 리포트를 작성하고 다음 개선 작업과 연결합니다.
- [ ] 개선 전후 평가 결과를 같은 기준으로 비교합니다.

### 4. 평가 자동화 및 개선활동 체계화

- [ ] LLM Judge 평가를 주기적으로 수행합니다.
- [ ] 평가 결과를 자동으로 집계하는 구조를 정리합니다.
- [ ] Dify 또는 Workflow 기반 테스트 환경을 안정화합니다.
- [ ] 외부 품질개선 툴 활용 결과를 내부 프로세스에 반영합니다.
- [ ] 주차별 개선 내역과 결과를 동일 포맷으로 기록합니다.

### 5. 워크플로우 및 구조 개선

- [ ] Workflow 세부 개발을 완료합니다.
- [ ] 확장성을 고려한 프롬프트 구조로 재정비합니다.
- [ ] 요약 생성, 후처리 분류, 평가 단계를 분리해 점검합니다.
- [ ] 요약 대상별로 공통 구조와 예외 규칙을 정리합니다.

## 최종 개선활동

### 데이터 정비

- 학습 및 검증 데이터 수급
- 민원/개선 데이터 추가
- 요약 기준과 정답 판정 기준 정리
- 유형별 베이스라인 재평가

### 프롬프트 개선

- 프롬프트 구조 개편
- 프롬프트 개선 포인트 반영
- 요약 기준 강화
- Few-shot 및 규칙 보강
- 키워드 추출 로직 보강
- 대/중/소 분류 프롬프트 개선

### 오류 분석

- 오류 유형 분석 수행
- 오답/과답/누락 데이터 분석
- 정답 가능 여부와 정답 변경 필요 케이스 검토
- false 데이터 분석 및 고도화
- 오류 리포트 작성

### 평가 및 자동화

- RAGAS 기반 정량 평가
- LLM Judge 평가 실시
- 평가 데이터 분석
- Dify 테스트 환경 및 Workflow 개선활동 안정화
- 외부 품질개선 툴 결과 반영

### 개선활동 관리

- 주차별 개선 로그 표준화
- 개선 전후 비교 결과 관리
- 유형별 개선 이력 누적
- 반복적으로 발생하는 오류 패턴 추적

## 제외된 항목

- 고객 발화 요약, 상담사 발화 요약, 상담 내용 요약 등 개별 문서 유형 이름
- 동일한 의미를 반복하는 일정 표현
- 주차별 중복 메모
