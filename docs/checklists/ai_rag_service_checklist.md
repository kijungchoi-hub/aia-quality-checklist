# AI RAG Service Checklist

참조 문서: `docs/checklists/answer_quality_checklist_final.md`

정리 원칙:
- 특정 도메인, 문서유형, 데이터셋 이름 제거
- 일반적인 AI + RAG 서비스에 공통으로 적용 가능한 항목만 유지
- 구현, 평가, 운영, 보안, 개선 루프를 모두 포함

## 1. 데이터 및 지식베이스 준비

- [ ] 수집 대상 문서와 데이터 소스를 정의한다.
- [ ] 최신성, 신뢰도, 접근권한 기준으로 데이터 소스를 분류한다.
- [ ] 원문, 요약, 메타데이터, 출처 정보를 함께 저장한다.
- [ ] 문서 청크 기준을 정의하고 일관되게 적용한다.
- [ ] 중복 문서, 오래된 문서, 충돌 문서를 정리한다.
- [ ] 색인 누락, 메타 누락, 청크 오매핑 같은 데이터 품질 이슈를 추적한다.
- [ ] 버전 변경 시 재색인 정책을 정의한다.

## 2. 검색 품질

- [ ] 기본 검색 전략을 정의한다.
- [ ] Lexical, Vector, Hybrid 검색을 비교한다.
- [ ] BM25, Dense Retrieval, Reranker 조합을 검증한다.
- [ ] Top-K, Score Threshold, Weight 조합을 최적화한다.
- [ ] Query Expansion 또는 Query Rewrite 적용 여부를 검토한다.
- [ ] 정답 문서가 상위 랭킹에 진입하는지 점검한다.
- [ ] 검색 실패 원인을 색인, 메타, 쿼리, 랭킹 단계로 분리한다.
- [ ] 검색 지연시간과 비용을 함께 측정한다.

## 3. 컨텍스트 구성

- [ ] 검색 결과를 그대로 넣을지, 정제된 컨텍스트만 넣을지 결정한다.
- [ ] Context Refinement 적용 여부를 검토한다.
- [ ] 상위 1~N개 컨텍스트만 사용하는 방식의 성능을 비교한다.
- [ ] 중복 청크, 노이즈 청크, 충돌 청크 제거 규칙을 정의한다.
- [ ] 서로 다른 문서 간 내용 충돌 시 우선순위 정책을 정한다.
- [ ] 근거 문서와 답변 문맥의 연결 가능성을 검증한다.

## 4. 프롬프트 및 생성 품질

- [ ] 시스템 프롬프트와 태스크 프롬프트의 역할을 분리한다.
- [ ] 답변 생성 프롬프트의 목표, 제약, 출력 형식을 명확히 정의한다.
- [ ] Hallucination 억제 규칙을 추가한다.
- [ ] 정보 부족 시 안전하게 거절하거나 보류하는 문구를 정의한다.
- [ ] Few-shot, 규칙, Anchoring, Routing 적용 여부를 검토한다.
- [ ] Prompt 축소, 단순화, 구조화 실험 결과를 비교한다.
- [ ] 검색 성공 후 생성 실패 케이스를 별도 관리한다.
- [ ] 답변과 함께 근거, 출처, 신뢰 수준을 제공할지 결정한다.

## 5. 모델 선택 및 튜닝

- [ ] 기본 생성 모델 후보를 비교한다.
- [ ] 성능, 비용, 지연시간, 운영 제약 조건을 함께 평가한다.
- [ ] SFT 적용 여부와 학습 데이터 기준을 정의한다.
- [ ] DPO, KTO, SimPO, IPO, ORPO 등 선호도 학습 전략을 비교한다.
- [ ] 튜닝 효과와 검색 개선 효과를 분리해서 해석한다.
- [ ] 튜닝 모델의 회귀 케이스를 별도로 추적한다.
- [ ] 운영 반영 전 안정성 검증 기준을 만든다.

## 6. 평가 체계

- [ ] 골든셋, 검증셋, 운영 샘플을 구분해 관리한다.
- [ ] 정답률, 오답률, 누락률, 과답률, 검색 매칭률 등 핵심 KPI를 정의한다.
- [ ] 자동 평가와 수동 검수를 함께 운영한다.
- [ ] RAGAS, LLM Judge, 휴먼 리뷰의 역할을 구분한다.
- [ ] 튜닝 전후, 프롬프트 변경 전후를 같은 기준으로 비교한다.
- [ ] 카테고리별, 사용자 유형별, 실패 유형별 결과를 분리해 본다.
- [ ] 평가 결과를 주기적으로 리포트한다.

## 7. 오류 분석 및 개선 루프

- [ ] 실패 케이스를 검색 실패와 생성 실패로 구분한다.
- [ ] 문서 미검색, 문서 오매칭, 근거 불일치, 답변 왜곡을 세분화한다.
- [ ] 오류 케이스 분석 결과를 프롬프트, 검색, 데이터, 모델 개선으로 연결한다.
- [ ] 반복적으로 발생하는 오류 패턴을 추적한다.
- [ ] 개선 전후 동일 샘플로 회귀 여부를 점검한다.
- [ ] 오류 분석 리포트를 표준 형식으로 남긴다.

## 8. 보안 및 정책

- [ ] 접근권한이 있는 문서만 검색되도록 권한 정책을 적용한다.
- [ ] 개인정보, 민감정보, 내부 전용 정보를 별도로 처리한다.
- [ ] 프롬프트 인젝션 및 데이터 유출 대응 정책을 만든다.
- [ ] 금지 답변, 법적 제한, 안전 정책을 정의한다.
- [ ] 로그에 저장해도 되는 정보와 제거해야 하는 정보를 구분한다.

## 9. 운영 및 관측성

- [ ] 요청 수, 지연시간, 토큰 사용량, 실패율을 수집한다.
- [ ] 검색 단계와 생성 단계의 병목을 구분해 측정한다.
- [ ] 버전별 프롬프트, 모델, 인덱스 변경 이력을 기록한다.
- [ ] 실험 로그, 결과, 회귀, 차주 계획을 같은 포맷으로 남긴다.
- [ ] 장애 대응 절차와 롤백 기준을 정의한다.
- [ ] 사용자 피드백을 수집하고 개선 작업과 연결한다.

## 10. 운영 전 최종 점검

- [ ] 대표 시나리오와 엣지 케이스를 모두 테스트한다.
- [ ] 잘못된 근거 인용, 자신감 높은 오답, 정보 누락을 확인한다.
- [ ] 비용과 지연시간이 목표 범위 안에 있는지 확인한다.
- [ ] 보안, 정책, 감사 로그 요구사항을 충족하는지 점검한다.
- [ ] 운영 반영 후 모니터링 항목과 책임자를 지정한다.

- 데이터 수집, 정제, 중복 제거, 버전 관리
- 메타데이터 보강과 색인 품질 개선
- 골든셋 및 검증셋 관리

### 검색 개선

- Hybrid Search, BM25, Dense Retrieval, Reranker 적용
- Top-K, Weight, Threshold 최적화
- Query Expansion 및 Query Rewrite 실험

### 생성 개선

- 프롬프트 구조 개선
- Hallucination 억제 규칙 반영
- 근거 기반 답변 형식 정교화
- Few-shot, Routing, Anchoring 실험

### 평가 및 개선 루프

- 자동 평가와 수동 평가 병행
- 오류 유형 분류 및 리포트 작성
- 회귀 테스트와 반복 개선

### 운영 및 관리

- 로그, 비용, 지연시간 모니터링
- 버전 이력 관리
- 보안 정책과 접근권한 관리
- 사용자 피드백 반영

## Appendix. 전문용어 설명

### RAG

- Retrieval-Augmented Generation의 약자다.
- 모델이 답변을 생성하기 전에 외부 문서나 지식베이스에서 관련 정보를 검색해 함께 활용하는 방식이다.
- 단순 LLM 답변보다 최신 정보 반영, 근거 제시, 내부 문서 기반 응답에 유리하다.

### Lexical Search

- 사용자가 입력한 단어와 문서에 포함된 단어를 직접 비교하는 전통적인 검색 방식이다.
- 키워드가 정확히 일치하거나 유사한 철자를 포함한 문서를 잘 찾는다.
- 대표적으로 BM25 같은 알고리즘이 사용된다.

### Vector Search

- 문장이나 문서를 임베딩 벡터로 변환한 뒤, 벡터 간 거리를 비교해 의미적으로 유사한 문서를 찾는 방식이다.
- 정확히 같은 단어가 없어도 의미가 비슷하면 검색될 수 있다.
- 표현이 다양하거나 질문이 우회적으로 들어오는 환경에서 유리하다.

### Hybrid Search

- Lexical Search와 Vector Search를 함께 사용하는 방식이다.
- 키워드 일치와 의미 유사도를 동시에 반영하므로, 한 가지 방식만 쓸 때보다 검색 안정성이 높아지는 경우가 많다.
- 운영에서는 두 방식의 가중치 조정이 중요하다.

### BM25

- 전통적인 정보검색 알고리즘 중 하나로, 문서 내 단어 빈도와 전체 문서 집합에서의 희소성을 함께 반영해 점수를 계산한다.
- 검색엔진에서 가장 널리 쓰이는 기본 랭킹 방식 중 하나다.
- 정확한 키워드 매칭이 중요한 업무 문서 검색에 여전히 강하다.

### Dense Retrieval

- 질문과 문서를 모두 임베딩 벡터로 바꾼 뒤, 의미적 유사도로 문서를 검색하는 기법이다.
- Vector Search의 핵심 검색 방식으로 이해하면 된다.
- 동의어, 문장 변형, 자연어 질문 처리에 강점이 있다.

### Reranker

- 1차 검색 결과를 다시 정렬하는 모델 또는 알고리즘이다.
- 1차 검색이 넓게 후보를 찾고, Reranker가 질문과 문서의 관련성을 더 정밀하게 판단해 상위 순위를 재조정한다.
- 검색 정확도를 높일 수 있지만 추가 비용과 지연시간이 발생한다.

### Top-K

- 검색 결과 중 상위 몇 개 문서를 사용할지 정하는 값이다.
- 예를 들어 Top-K가 5면 검색 결과 상위 5개만 답변 생성에 사용한다.
- 너무 작으면 필요한 근거를 놓칠 수 있고, 너무 크면 노이즈와 비용이 증가한다.

### Score Threshold

- 검색 점수가 일정 기준 이상인 문서만 채택하는 임계값이다.
- 낮게 잡으면 관련 없는 문서가 많이 들어오고, 높게 잡으면 필요한 문서까지 제외될 수 있다.
- Top-K와 함께 조정해야 한다.

### Query Expansion

- 원래 질문에 연관 키워드나 동의어를 추가해 검색 범위를 넓히는 기법이다.
- 예를 들어 약어, 공식 용어, 유사 표현을 함께 검색하도록 바꿔 검색 누락을 줄인다.

### Query Rewrite

- 사용자의 원래 질문을 검색에 더 적합한 형태로 다시 쓰는 기법이다.
- 길고 모호한 질문을 핵심 키워드 중심으로 정리하거나, 검색엔진이 잘 이해하는 구조로 바꾸는 데 사용한다.

### Context Refinement

- 검색된 문서를 그대로 넣지 않고, 답변에 필요한 부분만 정제해서 컨텍스트를 구성하는 작업이다.
- 중복 문장 제거, 불필요한 메타데이터 제거, 충돌 정보 분리 등이 여기에 포함된다.

### Hallucination

- 모델이 실제 근거 없이 그럴듯하지만 틀린 내용을 만들어내는 현상이다.
- RAG 환경에서도 검색은 성공했지만 생성 단계에서 왜곡이 생기면 Hallucination이 발생할 수 있다.
- 근거 제시, 답변 제한 규칙, 안전한 거절 정책으로 완화한다.

### Few-shot

- 프롬프트 안에 입력/출력 예시를 몇 개 함께 넣어 모델이 원하는 답변 형식을 따라가도록 유도하는 방식이다.
- 형식 일관성, 말투 통제, 판단 기준 전달에 유용하다.

### Anchoring

- 모델이 답변할 때 특정 기준 문장, 규칙, 근거 구조를 우선적으로 따르도록 고정하는 프롬프트 기법이다.
- 답변 스타일과 판단 기준의 흔들림을 줄이는 데 도움이 된다.

### Routing

- 질문 유형이나 난이도에 따라 서로 다른 프롬프트, 검색 전략, 모델, 처리 경로로 보내는 방식이다.
- 예를 들어 FAQ성 질문은 빠른 경로로, 복합 분석 질문은 정교한 경로로 분기할 수 있다.

### SFT

- Supervised Fine-Tuning의 약자다.
- 정답이 포함된 학습 데이터로 모델을 지도학습 방식으로 추가 학습시키는 방법이다.
- 특정 업무 말투, 응답 포맷, 도메인 지식 반영에 자주 쓰인다.

### DPO

- Direct Preference Optimization의 약자다.
- 사람이 더 선호하는 응답과 덜 선호하는 응답의 비교 데이터를 이용해 모델 선호를 학습시키는 방식이다.
- RLHF보다 구현이 단순한 편이라 최근 많이 사용된다.

### KTO

- Kahneman-Tversky Optimization의 약자다.
- 사람이 느끼는 이득과 손실의 비대칭성을 반영해 선호를 학습하려는 접근이다.
- 안전성이나 사용성처럼 단순 정답률만으로 측정하기 어려운 영역에서 검토된다.

### SimPO

- Simple Preference Optimization의 약자다.
- 선호도 학습을 더 단순한 목적함수로 처리하려는 계열의 기법이다.
- 구현 복잡도와 학습 안정성을 낮추려는 목적에서 비교된다.

### IPO

- Identity Preference Optimization의 약자다.
- 선호 학습 과정에서 기준 정책과의 관계를 더 안정적으로 유지하려는 방향의 방법론이다.
- 실험 설계에 따라 DPO 계열과 함께 비교 대상이 된다.

### ORPO

- Odds Ratio Preference Optimization의 약자다.
- 선호 응답과 비선호 응답의 차이를 확률 비율 관점에서 학습하는 기법이다.
- 선호도 학습 전략 비교 시 자주 함께 검토된다.

### RAGAS

- RAG 시스템 평가를 위한 대표적인 프레임워크다.
- 검색 적합성, 근거성, 정답 관련성 등 여러 평가 항목을 자동화해 측정할 수 있다.
- 휴먼 리뷰를 완전히 대체하기보다는 빠른 1차 품질 점검에 적합하다.

### LLM Judge

- 다른 LLM을 평가자로 사용해 답변 품질을 채점하거나 비교하는 방식이다.
- 대규모 자동 평가에 유용하지만, 평가 모델 자체의 편향과 일관성 문제를 함께 관리해야 한다.

### 골든셋

- 정답과 기대 결과가 명확히 정의된 대표 평가 데이터셋이다.
- 모델, 프롬프트, 검색 전략을 비교할 때 기준점 역할을 한다.

### 검증셋

- 모델이나 시스템 변경 후 성능을 확인하기 위한 평가용 데이터셋이다.
- 과적합을 줄이기 위해 학습 데이터와 분리해서 관리해야 한다.

### 회귀 테스트

- 변경 이후 기존에 잘 되던 기능이나 응답 품질이 나빠지지 않았는지 확인하는 테스트다.
- LLM 시스템에서는 동일 질문 세트에 대해 이전 버전과 결과를 비교하는 형태로 자주 운영한다.
